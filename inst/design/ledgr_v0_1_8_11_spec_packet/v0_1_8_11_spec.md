## 1. Thesis

v0.1.8.10 closed the v0.1.8.x single-core arc. The codebase now has a deeper
and better-ratified architecture: matrix-canonical strategy accessors,
fold-owned FIFO accounting, scoped public memory-backed B2 spot-FIFO opt-in,
parallel sweep dispatch, canonical JSON v2, durable event evidence, and
phase-separated benchmark reporting.

The governance worked. The problem is discoverability. Important decisions now
live across spec packets, RFC syntheses, final reviews, horizon entries,
benchmark notes, and contracts. A maintainer or agent can still recover the
truth, but doing so requires too much archaeology.

v0.1.8.11 is the entropy-management release before v0.1.9 target risk. It
should turn ratified decisions into a navigable architecture surface without
changing execution semantics.

## 2. Release Goals

v0.1.8.11 has nine planning goals:

1. Audit `contracts.md` for stale, missing, duplicated, or poorly organized
   contract language after v0.1.8.10.
2. Restructure or annotate `contracts.md` by surface where appropriate:
   execution spec, fold engine, output handlers, event evidence, accounting,
   strategy context, sweep, feature projection, persistence, documentation, and
   release gates.
3. Build a discoverable RFC / decision index that tells readers which accepted
   documents carry load-bearing decisions and which are historical scaffolding.
4. Populate ADRs for stabilized architectural decisions that currently live
   only in accepted RFCs or packet closeouts.
5. Grow the internal maintainer manual from the v0.1.8.8 skeleton into coherent
   articles. Manual articles may be reviewed in batches; partial completion is
   acceptable if the packet records a bounded remainder for the v0.1.9.x
   follow-on documentation slot. Priority order is execution / fold core,
   observability / determinism, sweep, snapshots/data, features, then benchmark
   methodology.
6. Refresh user-facing documentation for the post-v0.1.8.10 reality:
   who-ledgr-is-for, why R, strategy development, research workflow,
   reproducibility, sweep/B2 opt-in boundaries, and benchmark interpretation.
7. Add a plain-English financial research-software disclaimer surface and link
   it from the appropriate user-facing entry points.
8. Synthesize the v0.1.8.7-v0.1.8.10 performance arc internally: what got
   faster, where the measurements came from, how ledgr differs from peers, and
   which claims are intentionally not public marketing.
9. Housekeep `horizon.md`, the roadmap, design index, and agent notes so active
   and resolved entries reflect the v0.1.8.10 release and v0.1.8.11 planning
   state.

The release succeeds when a new contributor can answer "what is the current
architecture and where are the binding decisions?" without reading every RFC in
chronological order.

## 3. Binding Boundaries

### 3.1 Documentation Is Not Execution Scope

This packet does not authorize execution behavior changes. If the audit finds a
real contract bug, the correct outcome is a routed finding and a later ticket,
not an incidental implementation fix.

Allowed implementation side effects are narrow:

- stale-link repair;
- stale diagram correction;
- generated-documentation guard updates;
- doc test updates;
- typo / wording fixes that do not alter semantics;
- small helper scripts for documentation inventory or audit reports.

### 3.2 Contracts Audit Scope

The `contracts.md` audit should answer:

- Does every current public execution surface have a contract entry?
- Are removed or retired concepts still described as current behavior?
- Does the document distinguish canonical evidence from derived outputs?
- Does it reflect the post-v0.1.8.10 B2 scope guard:
  `compiled_accounting_model = NULL | "spot_fifo"`?
- Does it reflect that durable compiled integration remains deferred?
- Does it keep `ledgr_run()` and `ledgr_sweep()` on one shared fold core?
- Does it correctly separate strategy context, fill context, cost policy,
  target risk, liquidity, OMS, and future paper/live semantics?
- Is documentation-discovery behavior still current?

The audit may propose structural reorganization. It must not silently weaken
existing contracts.

### 3.3 RFCs Remain Authoritative

v0.1.8.11 synthesis should summarize and index accepted RFC decisions. It
should not rewrite accepted RFC text into replacement authority. The source
documents remain binding until superseded by a later spec, contract update,
ADR, or architecture note.

### 3.4 B2 Language Guard

B2 is a scoped spot-asset FIFO fill-batch accelerator for memory-backed sweeps.
Documentation must not describe it as:

- a general compiled fold core;
- default execution;
- durable compiled integration;
- a derivatives, margin, or non-spot accounting model;
- a replacement for event-sourced evidence.

### 3.5 Benchmark Language Guard

v0.1.8.x benchmark material is local, current-source, machine-specific
engineering evidence unless a future packet explicitly authorizes a public
claim. The v0.1.8.11 performance narrative is internal-facing first.

The teaching surface should live under `inst/design/manual/`, with raw
evidence, records, and measurement notes remaining under `dev/bench/`. The
manual article should link to the benchmark artifacts rather than copying every
number. The scoped narrative window is v0.1.8.7 through v0.1.8.10: Optimization
Round 2, parallel / diagnostic setup, v0.1.8.9 single-core optimization, and
v0.1.8.10 substrate / B2 closeout.

### 3.7 Manual Article Two-Layer Standard

A 2026-06-04 review of the first manual articles (`execution_fold_core`,
`performance_arc`) found them too synthesis-heavy: they restated contracts
and horizon entries without adding implementation depth beyond what the
public vignettes already cover. The corrective standard, binding for every
manual article in this packet:

Every manual article ships in two layers.

**Layer 1: Synthesis.** Orientation for a maintainer or agent forming a
mental model. Outcome-first opening, restated contract surfaces, scope
guards, "Where Next" close. This is what `execution_fold_core` and
`performance_arc` already do. Useful but insufficient on its own.

**Layer 2: Implementation Trace.** Depth aimed at a maintainer or agent who
needs to change code or debug behavior. The model is
`inst/design/manual/features.qmd`. Required
sub-sections per article:

- **Data structures.** Runtime shape (env / list / S3 class) of the objects
  the article covers, with slot listings and types. Example: "the per-pulse
  `ctx` env carries slots X, Y, Z with types A, B, C, populated at build
  time from sources P, Q, R."
- **Code anchors.** `file:line` references for every key boundary mentioned
  in the Synthesis layer. The reviewer spot-checks freshness; anchors that
  drift are repaired in the next batch that touches the article.
- **Lookup / dispatch mechanisms.** How the article's surfaces actually
  resolve at runtime: cache keys, hash algorithms, dispatch chains,
  registration mechanisms.
- **Edge cases.** What fails loudly vs silently. What is allowed-but-fragile.
  What is certified for parallel/resume equivalence vs not.
- **Hot/cold path distinction.** What runs per-pulse, per-instrument,
  per-feature, per-candidate, vs once at setup. The performance implications
  of design choices.
- **Concrete examples.** Small repr blocks showing actual R values where
  helpful. Not large data dumps.

The Implementation Trace section MUST NOT create new contracts. It describes
existing implementation. If the trace surfaces a contract gap, the gap is
routed to a follow-up ticket or RFC, not patched into the article.

The Implementation Trace section is read alongside the Synthesis section,
not instead of it. Both layers visible in the same article.

This standard applies to all v0.1.8.11 manual articles: the two existing
articles (`execution_fold_core`, `performance_arc_v0_1_8_x`) are retrofitted
via LDG-2546; the five new articles (LDG-2540, LDG-2541, LDG-2542, LDG-2543,
LDG-2545) author both layers from the start.

### 3.6 No Pass 2 Strategy Helpers

v0.1.8.11 may document the strategy accessor surface that shipped in
v0.1.8.10: `ctx$vec`, `ctx$idx()`, and
`ctx$vec$feature(feature_id)`. It must not add Pass 2 public strategy-helper
extensions. Rank-weight, inverse-vol, normalization, rebalance-band,
target-diagnostic, long-short, levered, hedged, cost-aware, or larger
declarative strategy constructors remain deferred to the v0.1.9.x helper
milestone or later RFCs.

### 3.7 Audit Artifact Homes

Review should confirm these paths before tickets are cut:

- `inst/design/ledgr_v0_1_8_11_spec_packet/contracts_audit.md` for the
  `contracts.md` audit report.
- `inst/design/ledgr_v0_1_8_11_spec_packet/generated_docs_audit.md` for the
  generated-docs / man-page audit and routed stale-language findings.
- `inst/design/rfc/README.md` or
  `inst/design/ledgr_v0_1_8_11_spec_packet/rfc_decision_index.md` for the RFC /
  decision index. The final home should be chosen before the ticket is cut; if
  the index is meant to stay evergreen, prefer `inst/design/rfc/README.md`.
- `inst/design/manual/performance_arc_v0_1_8_x.qmd` for the teaching narrative,
  with measurement records staying under `dev/bench/`.

The generated-docs / man-page audit output is a routed report: it should list
stale or suspect help-page language, classify each item as fix-now,
defer-with-reason, or no-action, and name the files/tickets that consume the
finding.
