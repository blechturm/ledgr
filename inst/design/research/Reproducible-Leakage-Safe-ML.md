# Architecting Reproducible Leakage-Safe ML in ledgr

## Bottom line

The strongest prior-art pattern is to separate the problem into three distinct contracts: a **point-in-time feature contract**, a **label-availability contract**, and a **model-lineage contract**. Feature stores such as Feast and Tecton formalize the first contract with per-row timestamps and point-in-time joins; L├│pez de Prado-style temporal validation formalizes the second with label intervals, purging, and embargoing; and MLflow, DVC, and W&B formalize the third with dataset digests, run metadata, artifact lineage, and versioned dependencies. A reproducibility-first backtester like ledgr should adopt those contracts explicitly, but keep the learner outside the engine. The engine should materialize a leakage-safe training panel, define when labels become legally usable, and record model/refit artifacts as content-addressed events; user-space adapters should own fitting, hyperparameter search, and prediction. ## Temporal discipline for panels and labels

In the L├│pez de Prado lineage, the key object is not just a timestamped row but a row with an **information interval**. In mlfinlabŌĆÖs PurgedKFold, `samples_info_sets.index` denotes when the information extraction started and the series value denotes when it ended; training observations whose label intervals overlap the test intervals are removed. Current combinatorial implementations such as skfolio define purging as removing training observations whose labels overlap in time with test labels, and define embargoing as excluding observations immediately after test observations because financial features can exhibit serial correlation. QlibŌĆÖs API exposes the same idea in a different form: `RollingGen` has `trunc_days` ŌĆ£to avoid future information leakage,ŌĆØ and `MultiHorizonGenBase` explicitly models `label_leak_n`, the number of future days required before a prediction-day label is complete. ŅłĆciteŅłéturn25view0Ņłéturn24search0Ņłéturn27search1Ņłéturn18view4Ņłü

For ledgr, that implies a clean rule for **training-panel export**. Each exported row should carry at least: `asset_id`, `asof_time`, a feature vector containing only values known by `asof_time`, a `label_interval_start`, a `label_interval_end`, and a `label_available_time`. For a fixed-horizon forward-return label with next-bar-open execution discipline, the natural label interval is the future return measurement window, and the natural availability time is the interval end plus any extra publication or processing lag. The engine should never hand an adapter a label row whose `label_available_time` exceeds the refit timestamp. This is the same pattern feature stores use for training-event tables: Feast requires an `event_timestamp` for each entity row and performs a point-in-time ŌĆ£last known good valueŌĆØ join at or before that timestamp; Tecton requires a timestamp per training sample and uses an AS OF join against `_effective_timestamp` to generate point-in-time-correct features. ŅłĆciteŅłéturn16view1Ņłéturn16view2Ņłéturn41view3Ņłéturn41view4Ņłü

That same structure gives you a precise purge/embargo computation. **Purging** should be driven by label overlap: if a training rowŌĆÖs label interval intersects any test rowŌĆÖs label interval, drop it. **Embargoing** should be driven by residual temporal dependence or delayed observability that is not already represented in the label interval itself: examples include execution latency, stale-close conventions, end-of-day data only becoming tradable next session, or features with serial dependence/publication lags near the split boundary. SkfolioŌĆÖs walk-forward documentation is unusually explicit here: `purged_size=0` is only appropriate when one can observe, compute, and execute within the same period with minimal latency, while `purged_size>=1` is the safer default when execution is delayed, such as daily-priced or illiquid assets. A good ledgr design is therefore to model **purge** from each rowŌĆÖs label interval and **embargo** from a separate, declared ŌĆ£latency / dependence bufferŌĆØ rather than collapsing both into a single horizon parameter. That recommendation is a synthesis, but it follows directly from the cited definitions. ŅłĆciteŅłéturn27search1Ņłéturn27search2Ņłéturn25view0Ņłéturn18view4Ņłü

What this catches is substantial but not complete. It directly addresses leakage from overlapping forward labels, serial dependence around split boundaries, and train/test contamination caused by temporally mis-specified preprocessing windows. It does **not** solve feature-engineering leakage from revised or future-known data, global scaling/imputation done before splitting, survivorship bias in the universe, nor research-loop leakage from repeated strategy mining and parameter search. WhiteŌĆÖs Reality Check and the Harvey-Liu line of work exist precisely because data reuse and multiple testing remain a false-discovery problem even after you have an out-of-sample split; Arnott, Harvey, and Markowitz make the same point in their protocol paper, warning that machine learning does not rescue finance from limited sample sizes or overfitting. ## Model artifacts and lineage

Serious MLOps systems treat a fitted model as the output of a **recipe**, not just a blob. MLflowŌĆÖs dataset tracking records dataset name, source, schema, profile, and a dataset digest, and `mlflow.log_input()` logs training, validation, and test datasets with explicit contexts. Its Model Registry then links model versions back to the run, datasets, and experiments that produced them. W&B Artifacts does the same with input/output artifacts and lineage graphs, including reference artifacts that can track externally stored files with metadata and checksums. DVC expresses reproducibility one level lower: `dvc repro` re-executes a dependency graph from `dvc.yaml`, while `dvc.lock` and the run cache identify realized pipeline states by combinations of dependencies, commands, and outputs. ŅłĆciteŅłéturn40view0Ņłéturn40view2Ņłéturn40view3Ņłéturn40view5Ņłéturn15view5Ņłéturn15view6Ņłéturn11search6Ņłéturn15view0Ņłéturn15view1Ņłéturn15view2Ņłü

For ledgr, the minimal auditable metadata set should therefore include more than a model file. At minimum, the artifact identity should cover the training-panel manifest or digest; the market-data snapshot hash; the feature manifest and feature code version; the label recipe including horizon, ranking method, and availability lag; the learner spec and hyperparameters; all seeds and thread-count / determinism settings; the train/validation/test temporal partitions plus purge/embargo policy; the preprocessing recipe and its fitted state; the package environment lockfile; and the serialized fitted object itself. In R, `renv::snapshot()` and `renv::restore()` are the right baseline for package-level reproducibility, because they record and later restore the project lockfile state, including repositories and exact package versions. ŅłĆciteŅłéturn39search0Ņłéturn39search1Ņłéturn39search2Ņłéturn39search6Ņłéturn39search8Ņłéturn40view0Ņłéturn40view5Ņłéturn15view5Ņłéturn15view2Ņłü

The crucial caveat is that **lineage is not the same thing as byte-reproducibility**. Ranger exposes a `seed` argument and a `num.threads` argument, which is exactly the sort of control you want to capture in a recipe hash. But official PyTorch guidance is blunter: completely reproducible results are not guaranteed across releases, commits, platforms, or even CPU versus GPU with identical seeds, although deterministic modes and explicit seeding reduce nondeterminism for a fixed environment. That means ledgr should differentiate between two standards: **reconstructable** and **bitwise reproducible**. The first is usually achievable with enough lineage; the second should be advertised only for a frozen environment, fixed package versions, fixed hardware/backend settings, and deterministic algorithms where available. ## Train predict refit patterns in current frameworks

The best positive example on the training side is Qlib. Its `DataHandlerLP` explicitly separates `infer_processors` from `learn_processors`, allows processors to be fitted on declared `fit_start_time` / `fit_end_time`, and can persist processed data to disk. Its rolling-task utilities expose expanding or sliding retraining schedules and include `trunc_days` to avoid future leakage, while the recorder subsystem logs parameters, metrics, objects, and artifacts and can sit on top of MLflow. This is close to the architecture ledgr wants: temporal windows are first-class, preprocessing can be fit on historical data only, and models/predictions are treated as artifacts. ŅłĆciteŅłéturn19view0Ņłéturn19view1Ņłéturn19view2Ņłéturn19view3Ņłéturn20view2Ņłéturn20view3Ņłéturn20view4Ņłéturn18view1Ņłéturn18view2Ņłéturn18view3Ņłü

The main R-native positive example is the tidymodels preprocessing discipline. `recipe()`, `prep()`, and `bake()` make a sharp distinction between estimating preprocessing parameters and applying them to new data. `step_normalize()` says explicitly that means and standard deviations are estimated from the training data supplied to `prep()` and then applied to new data with `bake()`. That is the exact discipline ledgr should want at the adapter boundary: the engine exports admissible history; the adapter `prep()`s on that history only; and the fitted recipe object is reused for pulse-time prediction on future rows. RsampleŌĆÖs time-based resamplers add sliding windows, though their official documentation positions `sliding_window()`, `sliding_index()`, and `sliding_period()` as flexible time-resampling tools rather than finance-specific purged CV tools. ŅłĆciteŅłéturn46search4Ņłéturn46search0Ņłéturn46search2Ņłü

The main R-native negative example is that this safety is **conventional, not structural**. If a user precomputes normalized/imputed features on the full dataset before resampling, the recipes discipline is bypassed. Some recipe steps are also dangerous at prediction boundaries: the documentation for row-affecting steps such as `step_sample()` warns that removing rows at `bake()` time can have unintended or problematic consequences and often should be done outside the recipe entirely. So the ledgr seam should not be ŌĆ£give me a tibble and trust the user.ŌĆØ It should be ŌĆ£give me a sealed training export and a sealed pulse export, and the adapter may only `prep()`/`fit()` on the former and `bake()`/`predict()` on the latter.ŌĆØ ŅłĆciteŅłéturn46search7Ņłéturn46search0Ņłéturn46search4Ņłü

Modeltime is helpful mainly as a lifecycle illustration, not as a direct template for panel ML. `modeltime_refit()` explicitly retrains models on new data while reusing training-time preprocessing steps, which is useful prior art for ŌĆ£refit the deployed model recipe on the latest admissible history.ŌĆØ But it is built for time-series forecasting tables, not for cross-sectional panels with per-row label availability, point-in-time universe membership, or overlap-aware purging. It should therefore be treated as a workflow hint, not as a finance-safe substrate. ŅłĆciteŅłéturn42search0Ņłéturn42search1Ņłéturn42search6Ņłü

Zipline, Alphalens, and Backtrader illustrate the opposite lesson: execution-time discipline does not automatically create training-time discipline. Zipline 3.0 stores multiple historical bundle ingestions and can rerun using an older bundle timestamp, which is genuinely useful for reproducibility; attached pipelines are computed each trading day and read in `before_trading_start()` or `handle_data()`. Alphalens is admirably explicit that prices for factor analysis must be the **next available tradable prices after factor observation**, not prices used in factor construction. Backtrader likewise states that the current bar has already happened and that standard market orders first execute on the next barŌĆÖs open; but its `cheat_on_open` and `cheat=True` timer modes intentionally allow same-day pre-open issuance and can blur the causal boundary if used carelessly. None of these frameworks gives you a safe training-panel abstraction by default. ## Cross-sectional panel targets and distributional ranking

The most relevant academic template for ledgrŌĆÖs motivating example is Gu, Kelly, and XiuŌĆÖs panel approach to the cross-section of stock returns. They frame expected-return measurement as a panel prediction problem, use a large stock-by-month panel with many characteristics, maintain temporal ordering by separating training, validation, and testing periods, and refit yearly rather than every month in their main implementation. They evaluate stock-level forecasts partly through portfolio sorts, including out-of-sample long-short decile spreads formed on model forecasts. That is conceptually very close to ŌĆ£predict score, then rank, then long top / short bottom.ŌĆØ ŅłĆciteŅłéturn33view0Ņłéturn34view0Ņłéturn34view2Ņłü

For your proposed target construction, the leakage-safe version is straightforward in principle and should be structural in code. First compute a forward return from a prediction-time convention that matches execution semantics. Then, for each date, transform those fully realized forward returns into a within-date rank or percentile on the **date-t eligible universe as it existed at t**, not on todayŌĆÖs survivors. The rank label is still forward-looking and therefore belongs only in the training substrate; its `label_available_time` is the time when all returns needed for that dateŌĆÖs cross-sectional rank are known. Gu, Kelly, and XiuŌĆÖs insistence on using a very broad CRSP universe and avoiding some common filters because of sample-selection concerns is a useful reminder here: the biggest rank-target leakage in practice is often not formulaic lookahead but anachronistic universe construction. ŅłĆciteŅłéturn35view1Ņłéturn35view3Ņłéturn29view0Ņłü

Quantile regression forests are a reasonable model-agnostic target interface for this use case. Ranger documents `quantreg = TRUE` and prediction with `type = "quantiles"`; it also notes that `keep.inbag = TRUE` is needed for out-of-bag quantile prediction. The older `quantregForest` package is even more explicit about the adapter possibilities: besides quantiles, `predict()` can return arbitrary functionals of the conditional distribution, including standard deviation, samples, or empirical-CDF calls via a custom `what` function. That means an adapter can naturally emit either a predicted median rank or an estimated probability such as `P(rank >= 0.8)` without any engine-side knowledge of how the learner works. ŅłĆciteŅłéturn36view0Ņłéturn36view1Ņłéturn37view0Ņłéturn37view1Ņłéturn36view3Ņłü

The practical pitfalls are methodological, not algorithmic. A within-date percentile target is **relative to the dateŌĆÖs cross section**, so its meaning shifts when the universe composition or cross-sectional dispersion changes; thin or ragged dates can make the label unstable; ties and missing-label rows need an explicit deterministic policy; and a predicted rank is not a calibrated expected return unless the adapter builds that mapping itself. Those issues are not arguments against the target, but they are reasons to store the universe snapshot, tie-breaking rule, eligibility filter, and ranking recipe as part of the label artifact rather than treating ŌĆ£rankŌĆØ as a self-explanatory column. ## Recommendations for ledgr

**Leakage-safe training-panel export.** Adopt a first-class **training-event table**. Each row should be keyed by `(asset_id, asof_time, row_hash)` and include feature values, feature manifest hash, universe membership hash, and time metadata (`asof_time`, `label_interval_start`, `label_interval_end`, `label_available_time`). Generate it from sealed data snapshots exactly once per export spec, and content-hash the export manifest. This is the same basic shape used by Feast and Tecton when they join features onto per-sample entity/timestamp tables and by MLflow when it tracks dataset inputs with digest and source metadata. Avoid allowing adapters to pull raw market data and construct their own historical matrices ad hoc, because that puts point-in-time correctness back into user convention. ŅłĆciteŅłéturn16view1Ņłéturn16view2Ņłéturn41view2Ņłéturn41view3Ņłéturn40view0Ņłéturn40view3Ņłü

**Forward-looking labels with horizon embargo.** Adopt labels as **training-only artifacts** with explicit availability timestamps. For a horizon label, the engine should structurally reject any row whose `label_available_time` is after the refit time; for validation, it should purge overlapping label intervals and apply a separately declared embargo buffer when execution latency or serial dependence requires it. QlibŌĆÖs `label_leak_n` is a good precedent for expressing ŌĆ£how long until this label is complete.ŌĆØ Avoid encoding the horizon only in documentation or naming conventions, and avoid deriving label eligibility from a loose ŌĆ£today minus NŌĆØ rule without recording the exact interval and lag semantics. ŅłĆciteŅłéturn18view4Ņłéturn27search1Ņłéturn27search2Ņłéturn25view0Ņłü

**Model-as-artifact identity.** Adopt a content-addressed **model recipe hash** over training-panel manifest hash, learner and adapter code version, hyperparameters, preprocessing recipe spec plus fitted recipe hash, seed(s), packages/environment lockfile, thread / deterministic flags, and evaluation/refit metadata. Store the serialized fitted object as an artifact and log the run lineage in the ledger as an event. MLflow, W&B, and DVC all converge on this basic lineage pattern, and `renv` is the obvious R-native way to freeze package dependencies. Avoid treating a model blob filename as the modelŌĆÖs identity. ŅłĆciteŅłéturn40view0Ņłéturn40view5Ņłéturn15view5Ņłéturn15view6Ņłéturn15view0Ņłéturn15view2Ņłéturn39search0Ņłéturn39search1Ņłéturn39search8Ņłü

**In-loop / walk-forward refit semantics.** Adopt a distinct **refit schedule** in the engine, separate from candidate selection. QlibŌĆÖs rolling tasks and ModeltimeŌĆÖs refit flow show the two primitives you need: a declared cadence and a recipe reused on new admissible history. In ledgr terms, a refit should be an engine event that consumes a sealed panel export through time `T`, produces a new model artifact, and activates that artifact for prediction on pulses `(T, next_refit]`. Avoid overloading your existing walk-forward candidate-selection feature to also mean model retraining, because selection-time leakage controls and deployment-time activation semantics are not the same thing. ŅłĆciteŅłéturn18view4Ņłéturn42search0Ņłéturn42search1Ņłü

**Model-agnostic adapter boundary and prediction hook.** Put the seam at **export / fit / predict**, not at learner internals. The engine should expose sealed training exports and sealed pulse-time feature payloads; the adapter should own `prep()` / `fit()` / `predict()` and return only predictions plus auditable metadata. TidymodelsŌĆÖ `prep()` / `bake()` split, FeastŌĆÖs ŌĆ£feature service per model version,ŌĆØ and TectonŌĆÖs use of the same feature service for both training-data generation and serving all point the same way: the engine should define the temporally legal data contract, and the learner should be an adapter behind that contract. ## Where prior art fails

Purged K-fold, embargo, and CPCV are extremely valuable, but their safety claims are often overstated. They address overlap-aware validation of labeled samples, not the full stack of leakage channels. They do not by themselves prevent revised-data leakage, global preprocessing leakage, point-in-time universe errors, or research-loop overfitting from repeated strategy mining. The older multiple-testing literature and the newer backtest-overfitting literature exist precisely because a mechanically correct split can still lead to false discoveries when the same history is mined repeatedly. ŅłĆciteŅłéturn27search1Ņłéturn30view0Ņłéturn31search2Ņłéturn31search0Ņłéturn31search1Ņłéturn31search3Ņłéturn31search4Ņłü

Feature stores solve a narrower problem than many users think. Feast and Tecton are excellent at per-row point-in-time joins and at keeping a model version tied to a feature-service definition, but they do not design the label for you, do not fix your universe definition, and do not stop you from using a logically impossible timestamp proxy. TectonŌĆÖs own docs say that if you do not have event records at the right granularity, you may need a proxy or a rearchitecture; that is exactly where trading-research leakage often begins. ŅłĆciteŅłéturn16view1Ņłéturn16view2Ņłéturn41view0Ņłéturn41view4Ņłü

Experiment trackers also oversell reproducibility unless the user is careful. MLflow, W&B, and DVC are very good at lineage, registry, run metadata, and pipeline state, but none of them magically makes a stochastic learner deterministic. Without seeds, thread settings, environment locks, and hardware/backend metadata, you get an auditable run history rather than a guaranteed byte-identical rebuild. PyTorchŌĆÖs reproducibility note makes that limitation explicit, and even CPU-versus-GPU parity is not guaranteed. ŅłĆciteŅłéturn40view5Ņłéturn15view5Ņłéturn15view6Ņłéturn15view0Ņłéturn15view2Ņłéturn45view0Ņłéturn45view1Ņłü

Backtesting frameworks usually protect the execution boundary better than the research boundary. ZiplineŌĆÖs historical bundles and BacktraderŌĆÖs default next-bar execution are good execution-time guardrails, while Alphalens is honest about next-available entry pricing. But the user can still contaminate studies with misaligned factor/pricing data, same-bar assumptions, or convenience modes such as `cheat_on_open`. In other words, these frameworks are helpful but not sufficient prior art for ledgrŌĆÖs goal of making it hard to fool yourself. ŅłĆciteŅłéturn22view2Ņłéturn22view3Ņłéturn22view0Ņłéturn22view1Ņłéturn23view0Ņłéturn23view1Ņłéturn23view2Ņłü

Tidymodels and modeltime get the ergonomics right but are not finance-safe by default. The `prep()` / `bake()` split is excellent, and modeltimeŌĆÖs refit semantics are useful, but neither package knows about label overlap, security lifetimes, as-reported fundamentals, or point-in-time panel membership. If users `prep()` once on the full panel and then resample, the leakage is entirely self-inflicted and entirely possible. Ledgr should therefore borrow their abstractions, not delegate its temporal guarantees to them. ŅłĆciteŅłéturn46search4Ņłéturn46search0Ņłéturn42search0Ņłéturn42search1Ņłü

## Open questions and limitations

The strongest evidence above is on point-in-time feature retrieval, artifact lineage, and refit lifecycle. The weaker area in currently accessible official docs is **sample uniqueness / return-attribution weighting**: the L├│pez de Prado literature discusses overlap dependence extensively, but contemporary framework documentation is much clearer on purging/embargoing/CPCV than on enforcing a canonical uniqueness-weighting scheme end-to-end. My recommendation for ledgr is therefore pragmatic: export optional overlap/concurrency weights derived from label intervals, but keep weighting policy in the adapter layer until you have a concrete use case that justifies engine-level standardization. That recommendation is a synthesis rather than a framework consensus. ŅłĆciteŅłéturn25view0Ņłéturn27search1Ņłéturn30view0Ņłü

A second limitation is that some implementation-specific claims above come from current documentation rather than source-code audits of every edge case. I am confident in the architectural patterns, but less confident in any claim that would require line-by-line verification of undocumented corner behavior. For ledgrŌĆÖs design, that is acceptable: the important part is to adopt the patterns that are consistently reinforced across papers and official docs, and to avoid inheriting any frameworkŌĆÖs undocumented assumptions wholesale.
