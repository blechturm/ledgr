# Validation Toolkit

This document assesses the current R ecosystem for a ledgr validation-toolkit RFC from an adapter-first perspective. The highest-confidence conclusion is that the R ecosystem is strong for return/risk analytics, serial-dependence-aware inference, variance-ratio testing, and general changepoint work, but still thin for production-grade selection-integrity tooling in the exact L├│pez de Prado/Bailey sense. ## Package landscape

**1. Current package landscape.** `PerformanceAnalytics` is the clear center of gravity for R-side performance and risk diagnostics. It is on CRAN now as version 2.1.0, published on 2026-04-11, maintained by Brian Peterson, and actively updated on GitHub in April 2026 ahead of that release. Its CRAN metadata and reverse dependency graph also matter politically: `PortfolioAnalytics` depends on it, and packages such as `portfolioBacktest`, `finlabR`, `tidyquant`, and others import or suggest it, so choosing it as ledgrÔÇÖs primary external evidence layer very clearly reads as ecosystem citizenship rather than competition. Function-wise, the current man-page index confirms `ProbSharpeRatio` and `MinTrackRecord`, but it does **not** show a direct `DeflatedSharpeRatio()` / `SharpeRatio.deflated()` entry or a `KRatio()` entry. In other words: PA already covers a meaningful subset of the BaileyÔÇôL├│pez de Prado Sharpe-inference lineage, but not DSR or KestnerÔÇÖs K-Ratio in current CRAN form. ţłÇciteţłéturn33view0ţłéturn31view0ţłéturn43search1ţłéturn43search0ţłü

`RPESE` is the strongest complement to `PerformanceAnalytics` for serial-dependence-aware inference. That is unusually well aligned with ledgrÔÇÖs deterministic, identity-bound requirements because the influence-function-based methods are parametric and deterministic, whereas bootstrap-based methods are available but optional. The package covers standard errors for Sharpe, Sortino, VaR, ES, and related measures, and `PerformanceAnalytics` now explicitly suggests `RPESE` and ships a ÔÇťStandard Errors for Risk and Performance EstimatorsÔÇŁ vignette; individual PA functions such as `DownsideSharpeRatio` also expose `SE` / `SE.control` hooks that route into `RPESE`-style controls. ## Algorithm gaps

**2. Algorithm gaps and what should be native.** The most important gap is the one the user suspected: `PerformanceAnalytics` gives ledgr `ProbSharpeRatio()` and `MinTrackRecord()`, but not a direct DSR implementation in current CRAN form. The closest production code I found is not in PA at all; it is in `quantstrat`, which exposes `deflatedSharpe` / `SharpeRatio.deflated` and `SharpeRatio.haircut`. That is actually good news architecturally, because DSR is mathematically small: the published BaileyÔÇôL├│pez de Prado formula needs an observed Sharpe ratio, number of trials, variance of trial Sharpe ratios, skewness, kurtosis, and the sample length. Since `PerformanceAnalytics` already computes Sharpe, skewness, kurtosis, and related moments, a ledgr-native DSR pure function layered above PA-shaped inputs should be modest. My engineering estimate is that the computational core is on the order of a few dozen lines, and a production-ready native implementation with input validation, docs, and tests is still small enough that reimplementing it is a lower-risk choice than depending on `quantstrat` just to get one formula. That estimate is an inference from the published DSR formula and the existing quantstrat implementation, not a measured line count from a formal code audit. ţłÇciteţłéturn24search7ţłéturn40search3ţłéturn43search1ţłü

The same logic applies to HarveyÔÇôLiu haircut-to-Sharpe. There **is** a current R implementation, but it lives in `quantstrat`, not in `PerformanceAnalytics`. So if the RFC wants haircut-Sharpe in scope, the right posture is probably the same as for DSR: cite the literature, inspect the quantstrat formula as a donor, then implement a small deterministic native function over ledgrÔÇÖs already-canonical return and trial metadata. That avoids dragging in the legacy stack while still standing on community work. ţłÇciteţłéturn39search1ţłéturn40search1ţłéturn40search4ţłü

For PBO and CSCV, the situation is more nuanced. R does have `pbo`, and the package really does implement CSCV-style PBO with a user-supplied performance function. But `pbo()` takes a `T x N` data frame of **returns**, slices it into CSCV subsets internally, and then evaluates candidate performance from those return series. That is a poor match for a ledgr architecture in which walk-forward may already have collapsed each candidate/fold to scalar scores. So even if `pbo` is used, it is best used as an optional adapter for a ÔÇťraw-return-matrixÔÇŁ pathway, not as the canonical engine for all selection-integrity diagnostics. If ledgr wants PBO over fold-level scalar scores, that is a different abstraction and should be implemented natively. ţłÇciteţłéturn8search0ţłéturn34view0ţłü

Purged K-fold CV, embargo, and CPCV are where the adapter-first strategy runs out of mature R ground. I did **not** find a clean, widely adopted, standalone CRAN package that is to purged/CPCV finance validation what `PerformanceAnalytics` is to return diagnostics. The closest R-native signal is `PortfolioTesteR`, whose reference manual includes a purged/embargoed K-fold helper for sequence models inside its broader framework. On the Python side, I did verify a documented `CombinatorialPurgedCV` object in `skfolio`, and I also found `pypbo`, which implements PBO/DSR/MinTRL-type functionality ÔÇö but `pypbo` is AGPL-3.0, which is a serious licensing constraint. For ledgr, the architectural conclusion is straightforward: purged K-fold, embargo, and any CPCV-over-ledgr-candidates logic should be native R in the core toolkit. The algorithms are fundamentally deterministic index construction and leakage exclusion, which are exactly the kinds of things worth owning if deterministic hashing, snapshot sealing, and business-objective reproducibility are hard requirements. Reticulate can remain a comparison route, not the core path. ţłÇciteţłéturn42search1ţłéturn41search5ţłéturn42search10ţłéturn23search4ţłéturn42search12ţłü

HRP is one place where the adapter-first rule should hold. `PortfolioAnalytics` is active and important, but I did not find an obvious current HRP surface in its publicly indexed docs, whereas `HierPortfolios` is a current CRAN package whose scope is explicitly hierarchical risk clustering and which directly exposes `HRP_Portfolio`. ## Architectural compatibility

**3. Architectural compatibility with ledgrÔÇÖs data model.** `PerformanceAnalytics` can absolutely be used with ledgrÔÇÖs canonical retained returns, but **not** in ledgrÔÇÖs long tibble shape directly. PA is explicit that it is primarily tested on return data, and individual functions such as `MinTrackRecord()` accept returns as `xts`, vectors, matrices, data frames, `timeSeries`, or `zoo`; rolling-chart helpers are likewise written around return series / matrices rather than long grouped tibbles. So the clean bridge is a one-way transformation from ledgrÔÇÖs canonical `period_return` field to an `xts` object (or, for some scalar functions, a matrix/data frame) indexed by `ts_utc`, one candidate at a time or one sweep-wide candidate matrix at a time. That is fully compatible with the ÔÇťno second canonical metrics pathÔÇŁ rule because the adapter consumes ledgrÔÇÖs already-computed retained returns, not raw fills or reconstructed positions. The bridge should sort by `(sweep_id, candidate_id, ts_utc)`, normalize all timestamps to UTC whole seconds, and pivot only the `period_return` column outward as needed. PAÔÇÖs appetite for regular return objects is a shape issue, not a methodological mismatch. ţłÇciteţłéturn33view0ţłéturn43search0ţłéturn25search16ţłü

The same is true, more sharply, for `pbo`: the shape mismatch is real and conceptually important. `pbo()` expects a `T x N` panel of raw returns, where `T` is samples per study and `N` is studies/candidates, then performs its own CSCV subdivision via the `s` argument and metric function `f`. It does **not** take a candidate-by-fold rank matrix or a precomputed matrix of fold-level scalar scores. That means ledgr has two coherent choices. Either ledgr preserves per-period return vectors for each candidate and uses `pbo` as an external classifier of overfit risk over those return streams, or ledgr accepts that its walk-forward engine produces fold-level summaries and implements its own PBO-like logic over those summaries. Trying to coerce fold scores back into `pbo` would be a conceptual mismatch, not merely a type conversion problem. ţłÇciteţłéturn8search0ţłéturn34view0ţłü

Determinism constraints cut across every adapter decision. `RPESE` is attractive precisely because its influence-function methods are deterministic, whereas its bootstrap modes are not unless seeds are explicit. `pbo` is deterministic as long as its parallel path is not allowed to introduce uncontrolled execution variability. `changepoint`, `vrtest`, `forecast::tsCV`, and `rsample` split generators are inherently friendlier to ledgrÔÇÖs ÔÇťsame inputs, same hashÔÇŁ model than `bcp`ÔÇÖs MCMC or `PortfolioAnalytics` metaheuristics such as random portfolios, differential evolution, particle swarm, and simulated annealing. In practical RFC terms: ledgr should treat ÔÇťalgorithmically deterministic by constructionÔÇŁ as a first-class selection criterion for adapters, not as an afterthought. ţłÇciteţłéturn24search1ţłéturn8search0ţłéturn21search2ţłéturn13search2ţłéturn15search2ţłéturn16search8ţłéturn16search7ţłü

## Licensing and package boundaries

**4. Licensing posture and boundary design.** The primary audited adapter packages split cleanly into two license groups. The permissive side includes `pbo` (MIT), `PortfolioTesteR` (MIT), and `finlabR` (MIT). The copyleft side includes `PerformanceAnalytics` (GPL-2 | GPL-3), `RPESE` (GPL-2 | GPL-3), `PortfolioAnalytics` (GPL-3), `HierPortfolios` (GPL-2), `PMwR` (GPL-3), and `vrtest` (GPL-2). On the Python side, the main verified caution flag is `pypbo`, which is AGPL-3.0 and therefore much more restrictive than the MIT/GPL mix in the R ecosystem. ţłÇciteţłéturn34view0ţłéturn42search1ţłéturn37view1ţłéturn33view0ţłéturn34view1ţłéturn34view2ţłéturn34view3ţłéturn35view0ţłéturn15search1ţłéturn23search4ţłü

For an RFC author, the practical implication is simple even without turning this into legal advice. If ledgr core needs to remain outside the GPL family, then the ÔÇťsmall-metric-native, optional-adapter-externalÔÇŁ strategy is the safest posture: implement tiny formulas such as DSR and K-Ratio natively; keep GPL adapters in optional boundaries; and avoid AGPL packages entirely unless the project consciously accepts those obligations. If, on the other hand, ledgr is already GPL-compatible, then adapting `PerformanceAnalytics` and `RPESE` becomes much easier politically and technically, because those are exactly the packages the rest of the modern R quant ecosystem is already composing around. The one package I would consciously avoid in any non-AGPL design is `pypbo`. ţłÇciteţłéturn33view0ţłéturn34view1ţłéturn23search4ţłü

I did **not** complete a primary-source audit of recent CRAN policy changes around `Suggests:` versus `Imports:` in this pass. ## Ecosystem citizenship

**5. Ecosystem citizenship and maintainer realities.** The strongest ÔÇťwho maintains whatÔÇŁ signal is that `PerformanceAnalytics` and `PortfolioAnalytics` are alive right now, under the stewardship of Brian Peterson / braverock, with CRAN publications in April 2026 and matching GitHub work in that same window. `RPESE` is also alive, with a December 2025 GitHub commit sequence and a January 2026 CRAN release. `PMwR` and `HierPortfolios` are similarly current enough to be credible adapters. `pbo` is still on CRAN, but it is clearly much quieter. `blotter` is not dead, but it is a legacy framework being kept working; `quantstrat` is more lightly active still; and `FinancialInstrument` is the hard stop, because it is both stale and archived from CRAN. ## Recommendations for the RFC seed

**6. Explicit recommendations.**

**Adapt these packages.** Ledgr should adapt `PerformanceAnalytics` first, because it is current, central, and already exposes the right core vocabulary for Sharpe/Sortino/Calmar/drawdown/rolling-performance style diagnostics. It should pair that with `RPESE` for deterministic, serial-dependence-aware standard errors and confidence-aware reporting. It should also adapt `vrtest` for variance-ratio / martingale-difference tests, and `changepoint` for deterministic regime-break detection when regime awareness is needed. If the RFC wants HRP in scope, use `HierPortfolios`, not a home-grown implementation. If the RFC wants extra practical reference points for streaks/drawdowns/trade summaries, `PMwR` is a healthier place to borrow ideas from than the `blotter` stack. ţłÇciteţłéturn33view0ţłéturn34view1ţłéturn15search2ţłéturn13search0ţłéturn34view3ţłéturn35view0ţłü

**Implement these algorithms natively.** Ledgr should implement DSR natively, because current CRAN `PerformanceAnalytics` stops at PSR/MinTrackRecord while `quantstrat`ÔÇÖs DSR implementation is trapped behind the wrong architectural stack. It should also implement K-Ratio natively, because there is no equally central adapter surface for it. Most importantly, ledgr should own purged K-fold CV, embargo handling, and any CPCV/CSCV logic that must work over ledgrÔÇÖs own candidate/fold objects, hashes, and snapshot semantics. Stable-region parameter analysis and Pardo-style robust-region diagnostics should also be native because the ecosystem does not currently offer a canonical package for them. Native here does **not** mean reinventing everything; it means implementing the small but load-bearing parts that must obey ledgrÔÇÖs deterministic data model. ţłÇciteţłéturn43search1ţłéturn24search7ţłéturn40search3ţłéturn5search11ţłéturn42search1ţłéturn42search10ţłü

**Defer or skip these methods/packages for the first RFC cycle.** Do not make `pbo` a foundational dependency even though it is valuable as a reference and optional adapter. Do not adopt `blotter` / `quantstrat` / `FinancialInstrument` as the state model for trade statistics. Do not make `bcp` or bootstrap-first inferential methods the default path unless seeds are explicit and included in the identity hash. Treat `PortfolioTesteR`, `finlabR`, and the new `RegimeChange` package as watchlist packages, not first-wave anchors. Treat reticulate-based Python wrapping as secondary to native R for core validation logic; the Python ecosystem is interesting here, but the verified `pypbo` implementation is AGPL and the main verified CPCV surface lives outside the R package governance story. ţłÇciteţłéturn34view0ţłéturn32view1ţłéturn32view2ţłéturn13search2ţłéturn42search1ţłéturn37view1ţłéturn34view6ţłéturn23search4ţłéturn42search10ţłü

**Adopt this licensing posture.** If ledgr core needs maximum downstream flexibility, keep the core toolkit native where formulas are small, and make GPL adapters optional or companion-package material. If ledgr core is already comfortable being GPL-compatible, then the adapter-first path around `PerformanceAnalytics` and `RPESE` is the cleanest and most socially aligned posture in the R finance ecosystem. Either way, avoid AGPL dependencies such as `pypbo` for the core RFC. ţłÇciteţłéturn33view0ţłéturn34view1ţłéturn23search4ţłü

**Keep the research output in one file for the RFC seed.** For this RFC stage, a single `Validation-Toolkit.md` is the right shape. The reason is not that the two literatures are identical; they are not. The reason is that the adapter choices, input-shape bridges, deterministic-sealing constraints, licensing posture, and ÔÇťexternal evidence versus canonical internal metricsÔÇŁ rule all cut across both literatures. A split into something like `Selection-Integrity-Diagnostics.md` and `Business-Objective-Constructor.md` would probably make sense later at implementation-spec time, once interfaces stabilize. But for the research and RFC-seed phase, a single document with two internal pillars is easier to reason about and less likely to duplicate the same architectural constraints twice. That is especially true because the most important recommendation from this pass is a hybrid, not a siloed, design: PA/RPESE adapters on one side, native selection-integrity machinery on the other, all fed by the same canonical retained-return snapshots. ţłÇciteţłéturn33view0ţłéturn34view1ţłéturn8search0ţłü
