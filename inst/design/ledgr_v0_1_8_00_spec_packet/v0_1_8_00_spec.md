## 1. Purpose

v0.1.8.00 is a prep cycle before v0.1.8 implementation work. It begins after
the v0.1.7.9 release gate has completed.

v0.1.7.9 shipped the execution-engine and public-documentation stabilization
needed before sweep design begins. The next feature release, v0.1.8, is expected
to introduce the fold-core/output-handler split and sweep mode. Before that
work starts, the design-document system needs to be made easier for agents and
human collaborators to navigate.

The current `inst/design/` directory is doing too much as a flat namespace. It
contains active contracts, roadmap material, architecture notes, RFCs, responses,
audits, spikes, operational playbooks, and historical versioned spec packets.
Those documents do not have the same authority. Treating them as a flat list
makes it easy for collaborators to over-weight stale audits or speculative RFCs,
and makes it harder to find the current load-bearing inputs for a ticket.

This cycle creates a governance layer for design documents. It should improve
discoverability and reduce context drift without changing package runtime
behavior.

The release has six goals:

1. **Create a canonical design index.** Add `inst/design/README.md` as the
   first file agents and humans read when orienting in the design system.
2. **Separate documents by operational role.** Keep contracts, roadmap, and
   foundational material easy to find, while moving RFCs, audits, spikes, and
   active architecture notes into clear subdirectories.
3. **Shorten the roadmap.** Keep the stable arc and active horizon in the
   roadmap, but stop duplicating completed packet detail or speculative DoDs for
   milestones several cycles out.
4. **Create a parking lot for future ideas.** Add `horizon.md` as a low-friction
   place for observations that are not yet roadmap commitments.
5. **Improve pkgdown site discoverability.** Fix the Articles navbar dropdown
   structure and verify that `build_llm_docs()` is generating LLM-optimized
   site artefacts.
6. **Execute the parallelism spike session.** Run the five platform spikes from
   `ledgr_parallelism_spike.md` and record findings before the v0.1.8 spec is
   opened. This is a deliberate scope extension: the spike work is v0.1.8 prep,
   not a governance deliverable, but it is included here to avoid a separate
   micro-cycle.

Goals 1-4 are intentionally non-runtime. They must not change execution
contracts, runner behavior, public APIs, package version metadata, or vignettes
beyond internal design references unless explicitly required by path updates.
Goal 5 is limited to `_pkgdown.yml` and CI configuration; it does not change
runtime R code. Goal 6 produces exploratory code in `dev/spikes/`, which is
excluded from the package build via `.Rbuildignore` and does not affect runtime
behavior.

## 2. Release Shape

v0.1.8.00 has nine coordinated tracks. Tracks A-G are the governance chain
(sequential). Tracks H and I are independent parallel tracks that can proceed
alongside the governance chain once Track A (baseline) is complete.

### Track A - RFC Disposition And Scope Baseline

Record the accepted governance model from:

- `inst/design/rfc_design_doc_governance.md`;
- `inst/design/rfc_design_doc_governance_response.md`.

The response accepts the RFC direction with two refinements:

- design documents need explicit authority labels, not just better folders;
- roadmap detail should cover the current milestone and next milestone, while
  future load-bearing constraints should be captured as short downstream notes.

This track owns final alignment before file moves begin.

### Track B - Design README And Authority Labels

Add `inst/design/README.md` as the canonical entry point for design documents.

The README must include:

- a cold-start reading order;
- role/authority definitions;
- a current-cycle pointer;
- task-specific entry points;
- a short warning that old spec packets are historical unless a task explicitly
  asks for them.

The cold-start reading order should be:

```text
1. contracts.md
2. ledgr_roadmap.md
3. active spec packet
4. only architecture/RFC/audit files relevant to the current ticket
```

Cross-cycle design documents should receive concise status headers where useful:

```markdown
**Status:** Active architecture input / Historical audit / Accepted RFC response
**Authority:** Contract / Proposal / Background / Operational
```

This track does not require retrofitting every historical spec packet.

### Track C - Directory Reorganization

Reorganize loose cross-cycle files under `inst/design/` by operational role.

Target shape:

```text
inst/design/
  README.md
  contracts.md
  ledgr_roadmap.md
  ledgr_design_document.md
  ledgr_design_philosophy.md
  model_routing.md
  ledgr_ux_decisions.md
  release_ci_playbook.md
  horizon.md

  adr/
  architecture/
  rfc/
  audits/
  spikes/

  ledgr_v0_1_8_00_spec_packet/
  ledgr_v0_1_7_9_spec_packet/
  ledgr_v0_1_7_8_spec_packet/
  ...
```

Proposed file placements:

| File | Target |
| --- | --- |
| `ledgr_v0_1_8_sweep_architecture.md` | `architecture/` |
| `ledgr_sweep_mode_ux.md` | `architecture/` |
| `ledgr_feature_map_ux.md` | `architecture/` |
| `sweep_mode_code_review.md` | `architecture/` |
| `rfc_cost_model_architecture.md` | `rfc/` |
| `rfc_cost_model_architecture_response.md` | `rfc/` |
| `rfc_design_doc_governance.md` | `rfc/` |
| `rfc_design_doc_governance_response.md` | `rfc/` |
| `execution_engine_audit.md` | `audits/` |
| `ledgr_parallelism_spike.md` | `spikes/` |

Root-level documents retained:

- `contracts.md`;
- `ledgr_roadmap.md`;
- `ledgr_design_document.md`;
- `ledgr_design_philosophy.md`;
- `model_routing.md`;
- `ledgr_ux_decisions.md`;
- `release_ci_playbook.md`;
- `horizon.md`.

All moves must use `git mv`. Path references must be updated in the same
commit as the moves.

### Track D - Horizon Parking Lot

Add `inst/design/horizon.md` as a non-binding design parking lot.

`horizon.md` is not a backlog, ticket queue, or roadmap commitment. It is a
place to preserve design observations that are not ready for roadmap promotion.

Required entry structure:

```markdown
### YYYY-MM-DD [area] Short title

Freeform note.
```

Area tags:

```text
execution, ux, data, risk, cost, research, infrastructure, adapters
```

Promoted or dropped items should move to a compact `Resolved` section with a
one-line disposition instead of disappearing silently.

### Track E - Roadmap Shortening

Refactor `inst/design/ledgr_roadmap.md` so it carries:

- stable vision and guiding principles;
- roadmap discipline;
- a milestone sequence table;
- one-line goals for completed milestones;
- full detail for the current and next milestone only;
- short downstream-constraint notes for future milestones when they affect the
  current architecture;
- links to authoritative spec packets for completed milestones.

Completed milestone detail belongs in the spec packet, not duplicated in the
roadmap.

Future milestones beyond the next one should keep only three to five intent
bullets unless a specific downstream constraint affects active design work.

### Track F - Agent And Path Reference Updates

Update agent-facing and design-facing path references after file moves.

At minimum:

- update `AGENTS.md` so it points agents at the design index and active packet;
- update references inside design documents affected by moves;
- update references in the new `inst/design/README.md`;
- run path greps before and after the moves.

This track should preserve local developer guidance and avoid unrelated edits.

### Track G - Verification And Closeout

Because this cycle changes design-document paths and agent context, verification
is mostly repository hygiene plus confirmation that the two parallel prep tracks
are complete:

- `git status --short` is clean except intentional local drafts before commit;
- path greps show no stale references to moved files;
- markdown files are readable and linked paths exist;
- no generated package artifacts are introduced;
- no package runtime files are changed unless explicitly justified.
- pkgdown site-discoverability evidence from Track H is recorded;
- all five parallelism spike outcomes from Track I are recorded.

No package test suite is required for this cycle unless runtime or package
documentation files are changed unexpectedly.

### Track H - pkgdown Site Discoverability

Improve discoverability of the published ledgr pkgdown site for human visitors
and LLM-based agents. This track is independent of the governance chain and can
run in parallel once Track A is complete.

Two changes:

1. **Articles navbar structure.** Set non-null `navbar:` labels in `_pkgdown.yml`
   so the Articles dropdown shows section headers (Start Here, Core Workflow,
   Design / Background) rather than a flat link to the index. This change is
   already applied to the working tree.

2. **LLM-optimized site artefacts.** pkgdown 2.1+ runs `build_llm_docs()` as
   part of `build_site()` by default. Verify that `LLMs.txt` and `.md` mirrors
   are generated at the site root after the next deployment. If absent, add an
   explicit call to the CI site-build step.

This track touches `_pkgdown.yml` and CI configuration only. It does not change
runtime R code, vignette content, or package APIs.

### Track I - Parallelism Spike Session

Execute the five platform spikes defined in `inst/design/ledgr_parallelism_spike.md`
and record findings. This track is a deliberate scope extension: the spike work
is a prerequisite for the v0.1.8 spec, not a governance deliverable, but it is
included in this cycle to avoid a separate micro-cycle.

The v0.1.8 spec must not be opened until all five spikes have recorded outcomes.

**Spike inventory:**

- SPIKE-1: mirai daemon lifecycle on Windows (native) and Ubuntu/WSL
- SPIKE-2: mirai task serialization - plain R objects, closures, environments
- SPIKE-3: mori cross-process serialization with mirai's NNG layer
- SPIKE-4: bar-data fetch strategy - pre-fetch vs. per-worker read-only DuckDB
- SPIKE-5: feature-cache cross-task survival in mirai daemon processes

**Code location:** `dev/spikes/0N_<name>/` - exploratory R scripts, not package
code. Add `dev/` to `.Rbuildignore`. Spike code does not need tests,
documentation, or R package conventions.

**Findings location:** `inst/design/spikes/ledgr_parallelism_spike.md`
(post-LDG-2003 path) or `inst/design/ledgr_parallelism_spike.md` if LDG-2003
has not yet run. Do not record spike outcomes in ticket files.

Each spike outcome must include: result (confirmed / refuted / inconclusive),
platform results (Windows, Ubuntu/WSL as applicable), and blocking implications
for v0.1.8 design decisions.

## 3. Hard Requirements

### R1 - No Runtime Behavior Change

v0.1.8.00 must not change:

- R package runtime behavior;
- runner semantics;
- snapshot semantics;
- strategy contexts;
- fill/accounting behavior;
- public APIs;
- exported functions;
- package version metadata.

This is design-document governance work only.

### R2 - Versioned Spec Packets Stay In Place

Existing `inst/design/ledgr_v*_spec_packet/` directories must remain in place.

They are archival records and are already path-stable. Moving them would create
review noise and break old references without improving active navigation.

### R3 - Root Load-Bearing Documents Stay Easy To Find

The following remain root-level under `inst/design/`:

- `README.md`;
- `contracts.md`;
- `ledgr_roadmap.md`;
- `ledgr_design_document.md`;
- `ledgr_design_philosophy.md`;
- `model_routing.md`;
- `ledgr_ux_decisions.md`;
- `release_ci_playbook.md`;
- `horizon.md`.

These are either authoritative, foundational, cross-cutting, or operational.

### R4 - Design README Must Name Authority Levels

`inst/design/README.md` must define the role categories used by the design
system:

- contract;
- roadmap;
- architecture input;
- RFC;
- audit/review;
- spike;
- spec packet;
- operational playbook;
- horizon note.

The README must identify what is load-bearing and what is background.

### R5 - Path Moves Must Preserve References

All file moves must use `git mv`.

Every moved file path must be grepped across the repository and updated where
the reference is still current. Stale references in historical spec packets may
remain only if they are clearly part of historical record text and not an active
instruction.

### R6 - Roadmap Detail Threshold

The roadmap must keep full detail for:

- the active prep/implementation milestone;
- the next milestone.

Milestones beyond the next should use intent bullets only, plus short
downstream-constraint notes when needed.

### R7 - `horizon.md` Must Stay Low Friction

`horizon.md` entries require only:

- date;
- area tag;
- short title;
- freeform note.

Do not add owner, due date, priority, acceptance criteria, or ticket status.
That would turn the file into a backlog, which is explicitly out of scope.

### R8 - Agent Instructions Must Point At The Design Index

`AGENTS.md` must make the new design index the first design-document entry
point.

It should point to:

- `inst/design/README.md`;
- `inst/design/contracts.md`;
- `inst/design/ledgr_roadmap.md`;
- the current active spec packet.

Historical packet links can remain only as examples or archival references.

### R9 - RFC And Response Pairs Stay Discoverable

RFCs and responses should move together into `inst/design/rfc/`.

The README should list active RFC inputs and accepted responses separately where
that distinction matters.

### R10 - Active Architecture Inputs Stay Together

Current v0.1.8 architecture inputs must be discoverable as a set.

At minimum, this includes:

- sweep architecture note;
- sweep UX note;
- sweep code review input;
- cost-model architecture response.

### R11 - No Hidden Governance Decisions

The file move and roadmap shortening must not silently change project scope.

If content is removed from the roadmap because it is speculative, it must be
either:

- preserved in a linked spec packet;
- preserved in `horizon.md`;
- preserved in a relevant architecture/RFC document;
- deliberately dropped with a note in the implementation summary.

## 4. Documentation Requirements

### D1 - Design README

`inst/design/README.md` should include:

- "Start here" section;
- current cycle pointer;
- task-specific entry points;
- role/authority table;
- active architecture inputs;
- RFCs;
- audits and spikes;
- historical spec packet guidance.

### D2 - Horizon

`inst/design/horizon.md` should include:

- a one-paragraph purpose statement;
- an "Open" section;
- a "Resolved" section;
- the area tag list;
- at least the known current horizon items that would otherwise live awkwardly
  in the roadmap.

Do not duplicate items that are already captured in the roadmap, architecture
notes, RFC responses, or active spec packets. `horizon.md` should hold
unplaced ideas, not repeat placed design decisions.

Candidate examples include:

- future `ledgr_snapshot_split()` helper;
- future persisted feature-series retrieval;
- public cost-model API after risk stabilizes.

### D3 - Roadmap

The shortened roadmap must still preserve:

- vision;
- guiding principles;
- roadmap discipline;
- v0.1.8 sweep/fold-core direction;
- downstream constraints needed for v0.1.8 architecture;
- completed milestone references.

It must not preserve full historical ticket-level detail for completed
milestones.

### D4 - AGENTS.md

`AGENTS.md` should align with the new design-document system.

It should teach agents to:

- start with `inst/design/README.md`;
- read the active spec packet for current work;
- preserve the one-execution-path invariant;
- avoid treating historical packets as active instructions unless explicitly
  requested.

## 6. Non-Goals

- No sweep implementation.
- No fold-core/output-handler refactor.
- No cost-model implementation.
- No risk-layer implementation.
- No package version bump.
- No public documentation overhaul.
- No changes to vignettes unless unavoidable path references require it.
- No migration of historical spec packet directories.
- No conversion of `horizon.md` into a backlog.
- No broad ADR rewrite.
- No attempt to resolve every future design idea during the governance pass.
- No promotion of spike code into package code (`R/`, `tests/`) within this
  cycle; spike findings inform v0.1.8 spec decisions, not implementation.
- No pkgdown vignette content changes; Track H is configuration only.

## 7. Suggested Ticket Families

The ticket split should keep file moves reviewable. The packet uses the standard
repo ticket files:

- `inst/design/ledgr_v0_1_8_00_spec_packet/v0_1_8_00_tickets.md`
- `inst/design/ledgr_v0_1_8_00_spec_packet/tickets.yml`

2. Add design README and horizon file.
3. Move cross-cycle design documents into role-based subdirectories, update path
   references, and update `AGENTS.md`.
4. Shorten roadmap and preserve future constraints.
5. Closeout verification and packet status after all required tracks complete.

Parallel tracks (independent after LDG-2001, both required before LDG-2005):

6. pkgdown site discoverability - navbar fix and `LLMs.txt` verification.
7. Parallelism spike session - all five spikes, findings recorded before v0.1.8
   spec opens.

The directory move, reference-update work, and `AGENTS.md` update must be one
ticket. A commit that moves files but leaves active references broken is not an
acceptable intermediate state. The roadmap shortening should be reviewed
separately because it changes content density rather than just paths.

## 8. Definition Of Done

v0.1.8.00 is complete when:

- the governance RFC and response are recorded and discoverable;
- `inst/design/README.md` exists and is the canonical design entry point;
- `inst/design/horizon.md` exists and is explicitly non-binding;
- loose cross-cycle documents are organized into role-based subdirectories;
- versioned spec packets remain in place;
- `ledgr_roadmap.md` is shortened to stable arc plus active horizon;
- future load-bearing constraints are preserved in roadmap notes, horizon notes,
  RFC responses, or architecture inputs;
- `AGENTS.md` points to the design index and active packet;
- path references to moved files are updated or intentionally left as
  historical text;
- `_pkgdown.yml` Articles sections have non-null navbar labels;
- `LLMs.txt` is present and populated at the pkgdown site root after deployment;
- all five parallelism spikes have recorded outcomes in the spike document;
- `dev/spikes/` exists and is in `.Rbuildignore`;
- no spike code is present under `R/` or `tests/`;
- no runtime/package behavior changes are included;
- the final diff is reviewable and the branch is ready to merge as v0.1.8 prep.
