# ledgr v0.1.0 Acceptance Tests (Given/When/Then)

These are milestone-level tests written in English to support peer review and agent management.
They must be translated into `testthat` integration tests where feasible.

---

## AT1 — Schema Initialization

**Given** an empty DuckDB database path  
**When** `ledgr_db_init(db_path)` is called  
**Then** all v0.1.0 tables exist with expected columns:
- runs, instruments, bars, features, ledger_events, equity_curve

---

## AT2 — Run Registration and Status

**Given** a valid config and preloaded instruments/bars in the DB  
**When** `ledgr_backtest_run(config)` starts  
**Then** a `runs` row is created with:
- status transitions: CREATED → RUNNING → DONE (or FAILED)
- config_json and config_hash stored
- data_hash stored
- error_msg NULL on success

---

## AT3 — Deterministic Replay (Core)

**Given** the same DB snapshot of `instruments` and `bars`  
**And** the same config (identical config_hash)  
**When** the backtest is executed twice into two separate run_ids  
**Then**
- `ledger_events` are identical (sorted by ts_utc, instrument_id, side, qty, price, fee, meta_json)
- `equity_curve` is identical within numeric tolerance
- `features` are identical

---

## AT4 — No-Lookahead in Features

**Given** a feature definition `F` and a past price series P  
**And** a future extension series Q where Q != P  
**When** feature value at time t is computed using P only  
**And** feature value at time t is computed using P+Q  
**Then** both results are equal within tolerance

---

## AT5 — Ledger Append-only

**Given** a completed run with `ledger_events` written  
**When** derived state is reconstructed via `ledgr_state_reconstruct(run_id)`  
**Then**
- positions and cash are derived solely from ledger + bars
- no other state table is required for correctness

---

## AT6 — Accounting Identities

**Given** any run  
**When** derived state is reconstructed  
**Then**
- I1: position qty matches cumulative signed fills per instrument
- I2: cash matches initial_cash + trade cashflows − fees
- I3: equity equals cash + positions_value (tolerance allowed)

---

## AT7 — Monotone Time

**Given** a completed run  
**When** equity_curve is queried ordered by ts_utc  
**Then** ts_utc is strictly increasing and covers the run’s pulse range

---

## AT8 — Resume Policy

**Cleanup requirement:** on resume, the engine must delete any previously written rows for the run tail (`ts_utc >= ts_resume`) in `ledger_events`, `features`, and `equity_curve` before continuing.

**Given** a run_id already present in `runs` with config_hash H  
**When** `ledgr_backtest_run()` is called with the same run_id and same H  
**Then** it resumes or reuses prior results without duplicating ledger rows

**And given** the same run_id but a different config_hash H2  
**When** backtest is started  
**Then** the system fails loud with a clear error

---

## AT9 — Minimal Strategy Compatibility (Non-ML)

**Given** a deterministic rule-based strategy (e.g. buy-and-hold)  
**When** backtest runs  
**Then** targets are accepted and translated to fills using the fill model

---

## AT10 — ML Strategy Compatibility (Deterministic Inference Only)

**Given** a strategy that loads a fixed artifact (hash recorded in config)  
**And** produces deterministic targets for the same inputs  
**When** backtest runs twice on same inputs  
**Then** results satisfy AT3 determinism and AT4 no-lookahead

*(No training; no stochastic inference.)*


## AT8b — Resume with Interrupted Run

**Given** a run that stopped mid-way (status=`RUNNING`, partial outputs present)
**When** `ledgr_backtest_run()` is called with the same run_id and config_hash
**Then** it resumes from the first not-yet-finalized pulse (after cleanup) and completes to status `DONE`.


## AT11 — Performance Sanity (Non-binding)

**Given** 10 instruments × 252 EOD bars
**When** backtest runs on reference hardware
**Then** it completes within a reasonable bound (suggested: < 60 seconds) and does not allocate unbounded memory.


## AT12 — Graceful Failure on Bad Data

**Given** bars containing an OHLC violation or duplicate timestamps
**When** backtest runs
**Then** run status becomes `FAILED` and `error_msg` contains actionable diagnostics.
