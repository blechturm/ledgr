# ledgr v0.1.0 Spec — Deterministic Backtest MVP (Core Spine)

**Status:** Draft for peer review (lock before coding)  
**Scope Gate:** This document is the *contract* for milestone v0.1.0. Anything not specified here is out of scope.

---

## 0. Goal

Deliver a **deterministic** end-to-end **EOD backtest** pipeline that produces:

- an **append-only event ledger** of fills/cashflows (`ledger_events`)
- **derived state** outputs (positions, cash, PnL, equity curve)
- **replayability**: same data snapshot + same config ⇒ identical outputs

This milestone establishes the *core spine*: **data → features → strategy → targets → fill sim → ledger → derived state**.

---

## 1. Non-Goals (Explicitly Out of Scope)

- Live trading / broker adapters
- Paper trading
- OMS lifecycle state machine (beyond a trivial “fill simulation”)
- Corporate actions handling (dividends/splits), survivorship bias, delistings
- Intraday bars, streaming, multiple pulses per day
- Advanced transaction cost models (impact, variable spreads)
- Portfolio optimization, advanced risk models (VaR/CVaR)
- ML training pipelines (only deterministic inference allowed in-strategy; see §11)
- UI / dashboards / Shiny
- Parallel bots (multi-process orchestration)

---

## 2. Terminology

- **Bar**: OHLCV record at a timestamp `ts_utc` (EOD semantics).
- **Pulse**: one decision tick in the backtest loop (EOD: one per trading day).
- **Targets**: desired end-of-pulse positions (qty) per instrument.
- **Fill**: simulated execution of an order intent, written as ledger event(s).
- **Ledger**: append-only event store; no updates-in-place.
- **Derived state**: positions/cash/PnL reconstructed from ledger + bars.
- **Snapshot**: immutable data input set; in v0.1.0, assumed preloaded in `bars`.

## 3. Hard Requirements (MUST)

### R1 Determinism
Given the same:
- `config_json`
- `bars` table contents for the run’s universe/time range

Then `ledger_events` and `equity_curve` MUST be identical across runs (byte-for-byte for JSON fields; numeric fields within tolerance where applicable).  
No dependence on system clock or external services during backtest.

### R2 No-Lookahead
At pulse time `t`, feature computation and strategy MUST only use bars with `ts_utc <= t_obs(t)`.

### R3 Append-only Ledger
`ledger_events` is append-only. Corrections must be modeled as compensating events (future versions).

### R4 Restart Safety (Backtest)
Backtest must be resumable from persisted DB state:
- **Resume cleanup rule (v0.1.0):** before resuming from pulse `ts_resume`, delete any previously written `ledger_events`, `features`, and `equity_curve` rows for that `run_id` with `ts_utc >= ts_resume` to avoid duplicate/alternate-reality tails.
- If a run is interrupted and re-run with same `run_id`, behavior MUST be defined:
  - v0.1.0 policy: **fail loud** if `run_id` exists and `config_hash` differs.
  - v0.1.0 policy: allow “resume” only if `run_id` exists and `config_hash` matches.

### R5 UTC Standardization
All stored timestamps are UTC; all internal time computations use UTC.

### R6 Minimal, Explicit Fill Model
Default fill: **next bar open** + deterministic costs (bps + per-trade fee).

### R7 Strict Schema Validation
Schema must be validated at runtime and via tests (tables + columns exist; types compatible).

## 4. Public API (v0.1.0)

Only the following functions are considered public (exported). Everything else is internal.

### 4.1 `ledgr_db_init()`
```r
ledgr_db_init <- function(db_path) -> DBIConnection
```
- Creates (or opens) DuckDB database at `db_path`
- Ensures schemas exist (idempotent)
- Returns a DBI connection

### 4.2 `ledgr_backtest_run()`
```r
ledgr_backtest_run <- function(config, con = NULL) -> list(run_id = character(1), db_path = character(1))
```
- `config` is a list (or JSON) validated against v0.1.0 config schema (see §5)
- If `con` is NULL, opens DuckDB at `config$db_path`
- Creates a new `run_id` unless `config$run_id` provided
- Executes backtest loop and writes outputs to DB
- Returns identifiers to locate results

### 4.3 `ledgr_state_reconstruct()`
```r
ledgr_state_reconstruct <- function(run_id, con) -> list(
  positions = data.frame,
  cash = data.frame,
  pnl = data.frame,
  equity_curve = data.frame
)
```
- Reconstructs state from `ledger_events` + `bars` for the run’s universe/time range

### 4.4 `ledgr_features_compute()` (optional public; may be internal)
```r
ledgr_features_compute <- function(run_id, con, feature_defs) -> invisible(TRUE)
```
- Computes and stores features deterministically with leakage guards

**Note:** If kept internal for v0.1.0, still must exist as internal module with tests.

## 5. Config Contract (v0.1.0)

Config is a list (or JSON) with the following required fields:

```yaml
run_id: <optional string>
db_path: <string path>
engine:
  seed: <integer>   # must be recorded; RNG locked even if not used
  tz: "UTC"
universe:
  instrument_ids: [<string>]
backtest:
  start_ts_utc: <ISO8601>
  end_ts_utc: <ISO8601>
  pulse: "EOD"
  initial_cash: <numeric>  # REQUIRED
fill_model:
  type: "next_open"
  spread_bps: <numeric>=0
  commission_fixed: <numeric>=0
features:
  enabled: <bool>
  defs: [ ... ]   # see §10
strategy:
  id: <string>
  params: <list>
```

### Config hashing
- `config_json`: canonical JSON with stable ordering
- `config_hash`: `digest::digest(config_json, algo="sha256")`

## 6. Data Contract (Inputs)

**Data ingestion is OUT OF SCOPE for v0.1.0.**
Bars and instruments must be **pre-loaded** into the DuckDB database by the user (or by test fixtures). v0.1.1 will introduce first-class data adapters/snapshotting.

### 6.1 Instruments table requirements
In v0.1.0, `instruments` must already contain rows for all `instrument_ids` in universe.

### 6.2 Bars table requirements
Bars must be present for the run’s instruments and time range.

**Data quality policy (v0.1.0): fail loud unless explicitly flagged as a gap.**
- OHLC violations → **ERROR**, fail the backtest.
- Duplicate timestamps per instrument → **ERROR**, fail the backtest.
- Missing expected trading days → represent explicitly using `gap_type != NONE` / `is_synthetic = TRUE` (if you choose to synthesize), otherwise **WARN** and continue. (No automatic synthesis in v0.1.0.)

Minimum bar requirements per instrument:
- at least 2 bars (so “next open” fill can be defined)
- OHLC sanity: `high >= max(open, close, low)` and `low <= min(open, close, high)`
- `ts_utc` strictly increasing per instrument (no duplicates)

**Data hash (v0.1.0 pragmatic):**
- Hash of (`instrument_ids`, `start_ts_utc`, `end_ts_utc`, plus a deterministic fingerprint of the bars subset ordered by `(instrument_id, ts_utc)`).
- **Floating-point stability for data_hash:** before hashing, numeric bar columns MUST be rounded to a fixed precision (v0.1.0: **8 decimal places**) to reduce cross-platform float representation drift.
- **Performance note:** implement hashing in a columnar/streaming manner where possible (avoid materializing gigantic R objects). Exact optimization is implementation-defined, but determinism is not optional.

## 8. Backtest Loop Semantics (EOD)

For each trading day `k` with timestamp `t = ts_utc(k)`:

- Observation time: `t_obs = t` (v0.1.0: bars already timestamped; no delay model)
- Strategy sees bars up to `t_obs`
- Strategy produces **targets** at `t_obs`
- Engine computes target deltas vs current derived positions
- Engine simulates fills at **next bar open** (`ts_next = next ts_utc`) using `open`:
  - buy fill price = `open_next * (1 + spread_bps/10000)`
  - sell fill price = `open_next * (1 - spread_bps/10000)`
  - commission applied deterministically: `commission_fixed` per fill event
- Ledger writes events at `ts_next` for fills (and optionally fee)

**Note:** v0.1.0 does not model partial fills, order aging, cancellations.

**Target translation rules (v0.1.0):**
- Strategies output **quantities** (`target_qty`) only (no weights in v0.1.0).
- `delta_qty = target_qty - current_position_qty`
  - `delta_qty > 0` → BUY `abs(delta_qty)`
  - `delta_qty < 0` → SELL `abs(delta_qty)`
  - `delta_qty = 0` → no order

**Termination policy (last bar):**
- If there is no `ts_next` (final pulse), **no fills can be simulated**.
- v0.1.0 behavior: generated targets on the final pulse are **discarded with a WARN** `LEDGR_LAST_BAR_NO_FILL`.

## 9. Invariants (Tested)

### I1 Position quantity consistency
For each instrument, derived position qty equals cumulative signed fill qty.

### I2 Cash consistency
Cash equals `initial_cash + sum(trade_cashflows) - sum(fees)`.

### I3 Equity identity
`equity == cash + positions_value` within tolerance.

### I4 Deterministic replay
Two runs with same config + same bars ⇒ identical ledger event set and equity curve.

### I5 No lookahead (features)
For each feature, the value at `ts` must not change if future bars are appended beyond `ts`.

### I6 Monotone time
Within a run: equity_curve `ts_utc` strictly increasing.

### I7 Stateless reconstruction (strategy)
If a run is resumed at pulse k (same `run_id` and `config_hash`), the strategy’s outputs from k onward must match a fresh run, **provided the strategy persists all required state exclusively via `state_update`**.

## 10. Feature Engine Contract (v0.1.0)

Feature definitions are a list of objects:

```r
feature_def <- list(
  name = "sma_20",
  requires_bars = 20L,
  stable_after = 20L,
  fn = function(bars_window, params) { ... },  # returns vector or scalar
  params = list(n = 20L)
)
```

Rules:
- Engine passes a **window** containing only bars up to current `ts`.
- Engine stores a **scalar** feature value at `ts` (last value if vector returned).
- If `current_index < requires_bars`, store `NA` (or omit row; choose one and document).
  - v0.1.0 policy: store `NA` rows explicitly for determinism.

Leakage test rule:
- For each feature_def, construct a synthetic series where future differs.
- Ensure `feature_at_t(past)` equals `feature_at_t(past+future)`.

## 11. Strategy Contract (ML and Non-ML)

### v0.1.0 Strategy Interface (R6)
Strategies are R6 objects implementing:

- `initialize(params, universe)`  
- `on_pulse(ctx)` returns `StrategyResult`

`ctx` contains:
- `ts_utc`
- `bars` (data available up to ts)
- `features` (optional)
- `positions` and `cash` (derived as of ts)
- `data_health` flags

`StrategyResult` contains:
- `targets`: data.frame(instrument_id, target_qty)
- `state_update`: optional JSON (persisted per pulse)

**ML strategies are allowed only as deterministic inference**:
- Model artifacts must be versioned and referenced in config.
- No training or stochastic inference during v0.1.0 backtests.

Non-ML strategies are first-class: rule-based, MA cross, rebalance, etc.
