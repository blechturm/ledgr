# ledgr v0.1.1 Spec — Data Ingestion & Snapshotting (Provenance Spine)

**Status:** Draft for peer review (lock before coding)  
**Scope Gate:** This document is the *contract* for milestone v0.1.1. Anything not specified here is out of scope.

---

## 0. Goal

v0.1.0 delivered the deterministic backtest “core spine”. v0.1.1 adds the missing **data provenance spine**:

- **Immutable, sealed snapshots** of market bars + instrument metadata
- **Reproducible replay**: a run references a snapshot and can be re-run months later
- **Tamper detection**: sealed snapshots are hash-verified before use

**Non-goals (v0.1.1):** corporate actions, PIT adjustments, OMS/live trading, real-time feeds, multi-asset semantics beyond “EOD bars + simple instrument metadata”.

---

## 1. Terms

- **Snapshot**: an immutable dataset bundle stored in DuckDB: bars + instruments + metadata.
- **CREATED**: snapshot exists but not sealed; rows may be written.
- **SEALED**: snapshot is immutable; hash computed and stored.
- **FAILED**: snapshot import/seal failed; error captured; no SEALED guarantee.

---

## 2. Hard Requirements

**R1. Immutability:** A snapshot with status `SEALED` must be unmodifiable via `ledgr` functions **and** by schema enforcement (best-effort triggers).  
**R2. Deterministic Hash:** `snapshot_hash` must be deterministic across runs, given identical imported data after normalization.  
**R3. Tamper Detection:** When a SEALED snapshot is used for backtests, `ledgr` must recompute and verify `snapshot_hash`; mismatch → fail loud with `LEDGR_SNAPSHOT_CORRUPTED`.  
**R4. Atomic Seal:** `ledgr_snapshot_seal()` must transition to `SEALED` and write `snapshot_hash` in a single DuckDB transaction as the final step.  
**R5. Subset Universes:** `config$universe$instrument_ids` may be a **subset** of snapshot instruments.  
**R6. Per-Instrument Coverage:** For any instrument in the universe, the backtest `[start_ts_utc, end_ts_utc]` range must be fully covered by snapshot bars for that instrument.  
**R7. CSV Contract:** CSV inputs must follow a strict format contract (§6.4).  
**R8. Numeric Stability:** OHLCV are **rounded to 8 decimals on import**, and the same rounded values are stored and hashed.

## 3. Public API (v0.1.1 additions)

### 3.1 Snapshot lifecycle

```r
ledgr_snapshot_create(con, snapshot_id = NULL, meta = list()) -> snapshot_id
ledgr_snapshot_import_bars_csv(con, snapshot_id, bars_csv_path,
                              instruments_csv_path = NULL,
                              auto_generate_instruments = TRUE,
                              validate = c("fail_fast", "none")) -> invisible(TRUE)
ledgr_snapshot_seal(con, snapshot_id) -> snapshot_hash
ledgr_snapshot_info(con, snapshot_id) -> data.frame
ledgr_snapshot_list(con, status = NULL) -> data.frame
```

If `snapshot_id` is NULL, ledgr generates:

`snapshot_{YYYYmmdd_HHMMSS}_{4-hex}`

Example: `snapshot_20251219_153045_a3f2`

### 3.2 Runner integration (v0.1.1 behavior change)

`ledgr_backtest_run(config, run_id = NULL)` (existing public API) gains:
- snapshot selection via `config$data$snapshot_id`
- mandatory `SEALED` hash re-verification before use

## 5. DuckDB Schema (v0.1.1 additions)

New tables (names are normative):

- `snapshots(snapshot_id PK, status, created_at_utc, sealed_at_utc, snapshot_hash, meta_json, error_msg)`
- `snapshot_instruments(snapshot_id, instrument_id, symbol, currency, asset_class, multiplier, tick_size, meta_json, PK(snapshot_id, instrument_id))`
- `snapshot_bars(snapshot_id, instrument_id, ts_utc, open, high, low, close, volume, PK(snapshot_id, instrument_id, ts_utc))`

**Status enum:** `CREATED | SEALED | FAILED`

### 5.1 Immutability enforcement (schema-level)

Best-effort triggers (DuckDB) to prevent INSERT/UPDATE/DELETE on `snapshot_*` rows for SEALED snapshots.
If triggers are unsupported in a given environment, code-level guards MUST still exist; tamper detection (R3) remains mandatory.

### Note on schema-level immutability enforcement (DuckDB)

DuckDB may not support row-level triggers / trigger-based write blocking in all builds/environments.
Therefore, **v0.1.1 MUST enforce immutability in code**:

- Every ledgr write path that touches snapshot tables MUST:
  1) read `snapshots.status`,
  2) fail-loud if status == "SEALED",
  3) never attempt partial writes.

Additionally, **tamper detection is mandatory** (see R3): on every use of a SEALED snapshot, recompute and verify `snapshot_hash`. If mismatch → `LEDGR_SNAPSHOT_CORRUPTED`.

## 6. CSV Format Contract

### 6.1 bars CSV (EOD only)



**Required columns:**
- `instrument_id` (string)
- `ts_utc` (ISO8601 UTC string: `YYYY-mm-ddTHH:MM:SSZ`)
- `open`, `high`, `low`, `close` (numeric)

**Optional columns:**
- `volume` (numeric; default `NA`)

**Rules:**
- encoding: UTF-8 (BOM allowed; if present, BOM is stripped)
- delimiter: comma `,`
- header row: required
- decimal separator: `.`
- `ts_utc` must be UTC with trailing `Z` (no offsets)
- `open/high/low/close/volume` are parsed as numeric then **rounded to 8 decimals** before storage
- OHLC validation at import/seal:
  - `high >= max(open, close, low)`
  - `low  <= min(open, close, high)`
  - any violation → fail loud
- duplicates on `(instrument_id, ts_utc)` → fail loud

### 6.2 instruments CSV (optional)

**Required columns if provided:**
- `instrument_id` (string)
- `symbol` (string)

**Optional columns (defaults):**
- `currency` (default `"USD"`)
- `asset_class` (default `"EQUITY"`)
- `multiplier` (default `1.0`)
- `tick_size` (default `0.01`)

### 6.3 instruments auto-generation

If `instruments_csv_path` is `NULL` and `auto_generate_instruments = TRUE`:
- instruments are generated from unique `bars.instrument_id`
- `symbol = instrument_id`, defaults applied

If `instruments_csv_path` is `NULL` and `auto_generate_instruments = FALSE`:
- fail loud.

### 6.4 Gap fields

**Out of scope for v0.1.1.** (No `gap_type` / `is_synthetic` in snapshot CSV contract.)

## 7. Snapshot Hash (Deterministic Fingerprint)

`snapshot_hash` must include a deterministic fingerprint of **both**:
- `snapshot_bars` rows
- `snapshot_instruments` rows

### 7.1 Canonical ordering

- Bars ordered by `(instrument_id, ts_utc)`
- Instruments ordered by `(instrument_id)`

### 7.2 Canonical numeric representation

All OHLCV are stored already rounded to 8 decimals. Hash uses the stored values.

### 7.3 Hash algorithm (streaming)

Implementation may chunk rows to avoid large in-memory objects:
- compute per-chunk sha256 over a stable text/binary representation
- final hash = sha256(concat(chunk_hashes))

#### Implementation notes (hashing must be streaming)

- The snapshot hash MUST be computed without materializing the full dataset in R.
- Use chunked fetching from DuckDB (cursor/fetch or LIMIT/OFFSET), with `chunk_size = 10_000` rows.
- Compute `sha256` per chunk, then `sha256(paste(chunk_hashes, collapse=""))` for the final fingerprint.
- Target memory usage: **< 500 MB** regardless of snapshot size.

## 8. Snapshot Seal

`ledgr_snapshot_seal(con, snapshot_id)`:

1. Validate snapshot exists and status is `CREATED` (else fail).
2. Validate referential integrity: all bars instrument_ids exist in snapshot_instruments.
3. Validate bars OHLC constraints and duplicates.
4. Compute `snapshot_hash` (bars + instruments).
5. In a **single DuckDB transaction**:
   - write `sealed_at_utc`
   - write `snapshot_hash`
   - set status to `SEALED`
6. Return `snapshot_hash`.

Empty snapshot policy (v0.1.1):

- If `snapshot_bars` has 0 rows at seal time → fail-loud with `LEDGR_SNAPSHOT_EMPTY`.
- Status remains "CREATED" (or becomes "FAILED" if you track seal failure explicitly), and `snapshot_hash` is not written.


If any step fails, snapshot status becomes `FAILED` and `error_msg` is recorded (transactional where possible).

---

## 9. Runner integration requirements

When `config$data$source == "snapshot"`:

1. Validate snapshot exists and status is `SEALED`.
2. Recompute hash and compare to stored `snapshot_hash`.
   - mismatch → abort `LEDGR_SNAPSHOT_CORRUPTED`.
3. Validate universe subset (R5).
4. Validate per-instrument coverage (R6).
5. Use `snapshot_bars` as the bars source for the run (no other sources in v0.1.1).

---

## 10. Invariants (v0.1.1)

- **I11 Snapshot referential integrity:** every `snapshot_bars.instrument_id` exists in `snapshot_instruments`.
- **I12 Snapshot seal immutability:** after SEALED, any attempt to modify snapshot_* through ledgr API fails.
- **I13 Tamper detection:** SEALED snapshot hash must match recomputed hash before any run uses it.
- **I14 Coverage:** for each instrument in the run universe, bars cover the full backtest range.

---

## 11. Migration / Compatibility

- v0.1.1 adds snapshot tables but does not change v0.1.0 run tables except adding optional `runs.snapshot_id` (if used).
- Runs created with v0.1.1 should store `snapshot_id` in `runs.config_json` and/or dedicated column if present.
