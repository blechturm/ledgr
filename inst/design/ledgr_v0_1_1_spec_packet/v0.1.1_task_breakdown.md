# ledgr v0.1.1 Task Breakdown — Data Ingestion & Snapshotting

This is the engineering plan for v0.1.1. Each ticket must include unit tests. Do not implement beyond scope.

------------------------------------------------------------------------

## Ticket 1.1 — Schema: snapshots tables + migrations

**Deliverable** - Extend `ledgr_create_schema()` / `ledgr_validate_schema()` to create and validate: - snapshots - snapshot_instruments - snapshot_bars - Add best-effort immutability triggers for SEALED snapshots (where supported).

**Tests** - create-from-scratch - idempotency - validator detects missing tables/columns/PKs - trigger blocks modifications when SEALED (skip test if triggers unsupported, but require code-level block)

------------------------------------------------------------------------

## Ticket 2.1 — CSV parser + format contract enforcement

**Deliverable** - Implement strict CSV reading helpers (internal) for bars + instruments: - required/optional columns - ts_utc ISO8601 Z-only parsing - numeric parsing with decimal '.' only - fail-fast option during import

**Tests** - good CSV accepted - missing columns rejected - bad timestamp rejected - duplicates rejected

------------------------------------------------------------------------

## Ticket 3.1 — Import pipeline (bars + instruments) with rounding

**Deliverable** - `ledgr_snapshot_import_bars_csv(...)` - requires snapshot status CREATED - stores OHLCV rounded to 8 decimals - inserts into snapshot_bars - inserts snapshot_instruments from file or auto-generation - enforces referential integrity and OHLC constraints (fail loud)

**Tests** - rounding stored == 8 decimals - auto-generate instruments works - import rejected when SEALED

------------------------------------------------------------------------

## Ticket 4.1 — Snapshot hash (bars + instruments)

**Deliverable** - Deterministic streaming hash for SEALED snapshots that includes: - fingerprint of snapshot_bars (ordered by instrument_id, ts_utc) - fingerprint of snapshot_instruments (ordered by instrument_id) - Store in snapshots.snapshot_hash.

**Tests** - same data → same hash - change bar value → hash changes - change instrument multiplier/tick_size → hash changes - ordering of inserts does not matter

“Done when”:

-   Uses DuckDB chunk fetch (chunk_size=10k)

-    Memory stays bounded (\<500MB)

-   Hash = sha256(concat(chunk_hashes))

-    hash includes **both** `snapshot_bars` and `snapshot_instruments` fingerprints (Gemini’s point)

------------------------------------------------------------------------

## Ticket 5.1 — Seal (atomic)

**Deliverable** - `ledgr_snapshot_seal(con, snapshot_id)` implementing §8 - Uses transaction; sets status SEALED and writes hash *last* - On error: status FAILED and error_msg

**Tests** - seal success path - simulated failure does not produce SEALED with partial metadata - second seal attempt fails

-   Seal runs inside a DB transaction

-    Status flips to SEALED **only after** snapshot_hash is written successfully

------------------------------------------------------------------------

## Ticket 6.1 — Runner integration: snapshot source + tamper verification

**Deliverable** - Extend `ledgr_validate_config()` to require `config$data$snapshot_id` for v0.1.1 runs - In `ledgr_backtest_run()`: - load snapshot bars for universe subset - re-verify snapshot_hash (tamper detection) - enforce per-instrument coverage for requested date range

**Tests** - subset universe allowed - tamper detection fails loud - coverage validation per instrument fails loud

------------------------------------------------------------------------

## Ticket 7.1 — Snapshot discovery APIs

**Deliverable** - `ledgr_snapshot_list(con, status=NULL)` - `ledgr_snapshot_info(con, snapshot_id)` with specified columns: - snapshot_id, status, created_at_utc, sealed_at_utc, snapshot_hash, bar_count, instrument_count, meta_json, error_msg

**Tests** - output columns present - status filter works

------------------------------------------------------------------------
