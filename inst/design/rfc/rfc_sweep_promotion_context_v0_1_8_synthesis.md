**Status:** Draft synthesis for final review.
**Date:** 2026-05-14
**Author:** Codex
**Inputs:**

## Purpose

This synthesis turns the promotion-context RFC and response into a concrete
v0.1.8 design candidate for final review.

The goal is:

```text
A run promoted from a sweep should be reproducible and selection-auditable.
```

v0.1.8 should not try to persist every sweep as a standalone replayable
artifact. Instead, a run created through `ledgr_promote()` should store a
compact, durable promotion context that explains the candidate superset and
selection view that led to the committed run.

## Accepted Direction

Accept the promotion-context idea for v0.1.8:

- `ledgr_sweep_results` remains an in-memory result object;
- `ledgr_candidate()` selects one promotion-ready candidate;
- `ledgr_promote()` commits the candidate through `ledgr_run()`;
- `ledgr_promote()` writes durable selection-audit metadata for the committed
  run;
- direct `ledgr_run()` calls do not have promotion context.

This gives ledgr a clear first milestone:

```text
v0.1.8 solves promoted-run selection audit.
Future work solves standalone sweep save/load/replay.
```

## Storage Decision

Use a dedicated table:

```text
run_promotion_context
```

Do not store promotion context as an opaque blob inside existing run metadata.
The dedicated table follows the existing store shape:

```text
runs
run_provenance
run_telemetry
run_tags
run_promotion_context
```

Recommended schema:

```sql
CREATE TABLE IF NOT EXISTS run_promotion_context (
  run_id                         TEXT NOT NULL PRIMARY KEY,
  promotion_context_version      TEXT NOT NULL,
  source                         TEXT NOT NULL,
  promoted_at_utc                TIMESTAMP NOT NULL,
  note                           TEXT,
  selected_candidate_json        TEXT NOT NULL,
  source_sweep_json              TEXT NOT NULL,
  source_candidate_summary_json  TEXT NOT NULL,
  selection_view_summary_json    TEXT
)
```

Schema migration:

- bump `ledgr_experiment_store_schema_version` from 106 to 107;
- old stores gain an empty `run_promotion_context` table;
- runs without rows in this table have no promotion context;
- `ledgr_promotion_context()` returns `NULL` for those runs.

The table should be included in schema validation.

## Context Version

Promotion context must be versioned:

```text
promotion_context_version = "ledgr_promotion_v1"
```

This versions the stored promotion-context structure. It is distinct from:

- `provenance_version`, which versions row-level sweep provenance objects;
- `seed_contract`, which versions seed derivation.

## Sweep ID

`sweep_id` must be defined before implementation.

Recommended contract:

- `ledgr_sweep()` generates a non-deterministic `sweep_id` at sweep start;
- `sweep_id` identifies a research event, not a deterministic input hash;
- two identical sweeps should have different `sweep_id` values;
- `sweep_id` is stored in `ledgr_sweep_results` attributes;
- `ledgr_candidate()` copies it into `sweep_meta`;
- `ledgr_promote()` writes it to `source_sweep_json`.

Do not use `.Random.seed` or ordinary strategy RNG to generate `sweep_id`.
v0.1.8 is tightening RNG boundaries, so sweep ID generation must not perturb
the seed stream used by strategy execution.

Acceptable implementation options:

- a UUID generated by a dependency if the dependency decision is accepted;
- an internal event ID using timestamp, process/session information, and a
  monotonic counter or entropy source that does not touch `.Random.seed`.

Avoid `sample.int()` unless it is isolated in a way that provably does not alter
the execution RNG stream.

## Candidate Summary Scope

The original RFC said `candidate_summary` should include all candidates. That is
right, but it needs a precise scope because the documented user workflow is:

```r
candidate <- train_results |>
  dplyr::filter(status == "DONE") |>
  dplyr::arrange(dplyr::desc(sharpe_ratio)) |>
  ledgr_candidate(1)
```

At `ledgr_candidate(1)` time, the input is a filtered/sorted view of the
original sweep. For selection audit, the promoted run should preserve both:

1. the original sweep candidate universe; and
2. the selection view from which the chosen row was selected.

Therefore promotion context stores two summaries:

```text
source_candidate_summary_json
selection_view_summary_json
```

### `source_candidate_summary_json`

This is the compact summary of all candidates in the original
`ledgr_sweep_results` object.

It answers:

```text
What was the full candidate universe?
Were there failed candidates?
What did the alternatives look like before filtering?
```

### `selection_view_summary_json`

This is the compact summary of the table passed to `ledgr_candidate()` after
user filtering/sorting.

It answers:

```text
What did the user actually select from?
Were failed candidates removed?
What ordering did the user create before selecting row 1?
```

If the input to `ledgr_candidate()` is the original unmodified sweep table, the
two summaries may be identical. The implementation may either store both
identical summaries or store `selection_view_summary_json = NULL` plus an
explicit marker that the selection view equals the source summary. The simpler
first implementation is to store both.

### Why Two Summaries

Only storing the source sweep hides the actual selection view.

Only storing the selection view can hide alternatives that were filtered out.

Storing both preserves the research decision without requiring ledgr to infer
the user's ranking/filtering intent.

## Promotion Note

`ledgr_promote()` should accept:

```r
note = NULL
```

The note is a user-authored explanation of selection intent, stored in the
`note` column of `run_promotion_context`.

Example:

```r
ledgr_promote(
  test_exp,
  candidate,
  run_id = "momentum_v1_test",
  note = "Top Sharpe from DONE candidates with at least 5 trades."
)
```

Do not try to store or infer ranking rules automatically. The user may have
filtered, ranked, or excluded candidates for reasons that are not recoverable
from a sorted table alone.

## JSON Serialization

Durable nested fields are stored as canonical JSON strings:

- `selected_candidate_json`;
- `source_sweep_json`;
- `source_candidate_summary_json`;
- `selection_view_summary_json`.

Do not try to store R list columns or R condition objects directly in DuckDB.

Before calling `canonical_json()`, convert candidate summaries into
JSON-compatible list-of-records. `canonical_json()` rejects data frames, so the
implementation must not pass a tibble directly.

Recommended candidate summary record fields:

```text
run_id
status
final_equity
total_return
annualized_return
volatility
sharpe_ratio
max_drawdown
n_trades
win_rate
avg_trade
time_in_market
execution_seed
params_json
provenance_json
n_warnings
warning_classes
error_class
error_msg
```

`params_json` and `provenance_json` are canonical JSON strings per row.

On read, helper APIs may parse these back into R lists for user convenience.
Round-trip behavior must be tested for the JSON value types supported by
`canonical_json()`.

## Warning Serialization

Live sweep results may carry warning condition objects in the `warnings` list
column.

Stored promotion context must not attempt to serialize full conditions.

Use:

```text
n_warnings
warning_classes
```

where `warning_classes` is a JSON array of unique warning class strings for the
candidate.

This preserves the audit questions:

```text
Were there warnings?
What kind of warnings?
```

without pretending full R conditions are durable data.

## Source Sweep JSON

`source_sweep_json` should include:

```text
sweep_id
snapshot_hash
strategy_hash
feature_union_hash
master_seed
seed_contract
evaluation_scope
n_candidates
```

It may include additional non-breaking fields under
`promotion_context_version = "ledgr_promotion_v1"` only if the version contract
states they are optional.

## Selected Candidate JSON

`selected_candidate_json` should include:

```text
run_id
params_json
execution_seed
provenance_json
status
key metrics
```

This duplicates one row from the summaries deliberately. It gives readers the
selected candidate directly without requiring them to scan the summary arrays.

## Read API

Ship both:

```r
ledgr_promotion_context(bt)
ledgr_run_promotion_context(exp, run_id)
```

Also expose the parsed context through:

```r
ledgr_run_info(... )$promotion_context
```

Behavior:

- promoted runs return a parsed promotion-context object/list;
- direct `ledgr_run()` runs return `NULL`;
- missing `run_promotion_context` table in old stores is treated as no context
  after schema migration/validation has run.

The helpers are read-only. They do not execute strategies, rerun sweeps, or
modify run state.

## Write Timing

`ledgr_promote()` writes promotion context only after `ledgr_run()` succeeds and
the committed run exists.

If `ledgr_run()` fails, no promotion context is written.

This avoids dangling promotion context rows for failed or partially committed
runs.

## Size Bound

Storing source and selection summaries duplicates compact sweep summaries across
promoted runs.

This is acceptable for v0.1.8:

- typical first-release sweeps are expected to be small to medium;
- the spec already warns when grids exceed 20 combinations without precompute;
- candidate summaries are much smaller than full event/equity artifacts;
- self-contained promoted runs are more valuable than premature sweep artifact
  registries.

If sweep sizes routinely exceed about 500 candidates, revisit this alongside
full sweep artifact persistence.

## Relationship To Future Sweep Artifacts

Promotion context is not full sweep persistence.

Future work, already parked in `horizon.md`, may add:

```r
ledgr_save_sweep()
ledgr_load_sweep()
ledgr_verify_sweep_sources()
```

That future feature can reduce duplication by storing one sweep artifact and
letting promoted runs reference it by `sweep_id`.

v0.1.8 should not block that future, but it should not implement it.

## Required Spec Changes If Accepted

If accepted, patch the v0.1.8 spec to:

1. Add `run_promotion_context` to the experiment-store schema with schema
   version 107.
2. Define `promotion_context_version = "ledgr_promotion_v1"`.
3. Define `sweep_id` generation at `ledgr_sweep()` start without touching
   strategy RNG state.
4. Add `note = NULL` to `ledgr_promote()`.
5. Define durable JSON fields:
   - `selected_candidate_json`;
   - `source_sweep_json`;
   - `source_candidate_summary_json`;
   - `selection_view_summary_json`.
6. Define warning serialization as `n_warnings` and `warning_classes`.
7. Define public read helpers:
   - `ledgr_promotion_context(bt)`;
   - `ledgr_run_promotion_context(exp, run_id)`;
   - `ledgr_run_info()$promotion_context`.
8. State that direct `ledgr_run()` runs have no promotion context.
9. State that full sweep artifact save/load is out of v0.1.8 scope.

## Required Tests If Accepted

Add tests for:

- schema migration creates `run_promotion_context` and bumps store schema to
  107;
- direct `ledgr_run()` has no promotion context;
- `ledgr_promote()` writes context only after successful run commit;
- `ledgr_promotion_context()` returns parsed context for promoted runs;
- `ledgr_promotion_context()` returns `NULL` for direct runs;
- `ledgr_run_promotion_context()` can read context by store/run ID;
- `ledgr_run_info()` includes optional `promotion_context`;
- `sweep_id` exists and does not perturb execution RNG state;
- source candidate summary includes all original sweep candidates;
- selection view summary reflects the filtered/sorted table passed to
  `ledgr_candidate()`;
- params round-trip through `params_json`;
- provenance round-trips through `provenance_json`;
- warning counts and warning class arrays are serialized correctly;
- failed candidates appear in source summary with status/error fields;
- promotion note is stored and recovered.

## Non-Goals

This synthesis does not require:

- automatic sweep artifact persistence;
- full sweep replay from disk;
- storing full ledger events or equity curves for every sweep candidate;
- recovering strategy or feature factory code from promotion context;
- inferring user ranking rules;
- storing R condition objects;
- a new execution path.

## Open Questions For Final Review

1. Is storing both `source_candidate_summary_json` and
   `selection_view_summary_json` worth the duplication, or should v0.1.8 store
   only one summary plus a `candidate_summary_scope` field?
2. Should `selection_view_summary_json` be nullable when it equals the source
   summary, or should the first implementation always store both for
   simplicity?
3. What is the preferred non-RNG `sweep_id` generator in this package: UUID
   dependency, timestamp/session/counter hash, or another internal helper?
4. Should `ledgr_promotion_context(bt)` be exported in v0.1.8, or should only
   store-level `ledgr_run_promotion_context()` and `ledgr_run_info()` expose
   the data?
5. Should `note` be plain text only, or should it support a small structured
   list in future versions?

## Recommendation

Accept promotion context for v0.1.8 with:

- dedicated `run_promotion_context` table;
- schema version 107;
- non-RNG `sweep_id`;
- `note = NULL` on `ledgr_promote()`;
- source and selection-view candidate summaries;
- canonical JSON serialization of nested records;
- warning counts/classes rather than condition objects;
- read helpers and `ledgr_run_info()` integration.

This keeps full sweep persistence deferred while making promoted runs
selection-auditable, which is the more important first user-facing contract.
